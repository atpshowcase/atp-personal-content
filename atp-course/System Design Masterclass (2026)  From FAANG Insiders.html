BAB A — System Design Interview Foundation

A.1 Understanding System Design Interviews

System design interviews are evaluated on clarity, structure, and the quality of your trade-off decisions. Interviewers are looking for engineers who can turn ambiguous requirements into a viable architecture, explain each choice, and adapt as new constraints appear.

  • How system design interviews are evaluated: Problem decomposition, correctness, scalability, reliability, and communication.
  • Communication and trade-off discussions: Always explain why you chose one approach over another, and show that you understand the downside as well as the upside.
  • Common interview structure: Requirements gathering, high-level design, component deep dive, and non-functional requirements.

A.2 System Design Framework

A reliable framework keeps your interview structured and memorable. Use the same sequence every time: gather requirements, estimate capacity, define APIs, sketch the high-level architecture, and then dive deeper into the riskiest parts.

  • Requirement gathering: Clarify functional and non-functional goals, user roles, and success metrics before drawing anything.
  • Capacity estimation: Validate that your design can support realistic load and growth scenarios.
  • API design: Define the key endpoints and contract shapes to anchor your architecture.
  • High-level design: Draw the main components and their interactions, then explain traffic flows.
  • Deep dive analysis: Pick one or two critical pieces—data partitioning, caching, consistency, availability—and examine them in detail.

A.3 Functional vs Non-Functional Requirements

Separating functional from non-functional requirements helps you avoid building the wrong system. Functional requirements tell you what the system must do. Non-functional requirements tell you how well it must do it.

  • Functional requirements: Features like user registration, post creation, search, redirection, or throttling.
  • Scalability requirements: Support for traffic growth, horizontal scaling, and elastic capacity.
  • Availability requirements: Uptime expectations, failure recovery, and graceful degradation strategies.
  • Latency expectations: Response time targets for user-facing operations and background processes.
This framework is reused in every system design problem.

BAB B — Capacity Estimation

B.1 DAU & MAU Estimation

Estimating active users is the first step in sizing any system. Use Daily Active Users and Monthly Active Users to convert product expectations into traffic projections.

  • Daily Active Users: The number of users who use the service in a single day.
  • Monthly Active Users: The number of unique users over 30 days, which helps estimate long-term retention and load patterns.
  • Growth assumptions: Model conservative, expected, and aggressive growth paths so you can choose architecture that handles the future, not just today.
Analogy: Forecasting Crowd Size for a Concert. If you only estimate people walking through the door today, you may be fine. But if you also estimate next month, next quarter, and next year, you can decide whether to build a small club or a stadium.

B.2 Throughput Calculation

Requests per second is the most important number in a system design interview. Break down user behavior into read and write operations, then map those operations to peak traffic volume.

  • Requests per second (RPS): Estimate the load during peak windows, not just average traffic.
  • Read-heavy vs write-heavy systems: Know which side dominates for your product and choose caches or write-optimized storage accordingly.
  • Peak traffic estimation: Use daily traffic, peak daily percentage, and burst factors to calculate the peak load that matters.
Analogy: Measuring Rush Hour on a Highway. Average speed across the whole day is not enough. You need to know how many cars show up at 6 PM, because that is when the road must still work.

B.3 Storage Estimation

Storage sizing keeps your architecture grounded. Estimate how much data is created each day, how long it must be retained, and whether long-term archive or active storage is required.

  • Daily storage growth: Multiply average object size by daily volume to understand new storage needs.
  • Yearly storage projection: Project growth over 12 months to decide whether you need tiered storage or lifecycle policies.
  • Media storage calculations: Account for large objects like images and videos separately from text or metadata.
Analogy: Managing a Warehouse. If you only count boxes arriving today, the shelves will overflow in a month. You need to know the rate of arrival and the lifetime of each box.

B.4 Memory Estimation

Memory sizing is key for cache layers and in-memory processing. Decide how much state must be stored in memory to meet latency goals without overcommitting expensive RAM.

  • Cache sizing: Estimate how much cached data is required to achieve the desired hit rate.
  • Working set calculations: Analyze the portion of data that is actively used at any given time.
Analogy: Your Desk Workspace. You only keep the files you're actively using on your desk. Everything else stays in the cabinet. If your desk is too full, you slow down.

B.5 Network Bandwidth

Bandwidth matters more than developers often realize. Estimate the traffic across each boundary in your architecture so you avoid choosing a design that bottlenecks on network capacity.

  • Upload traffic: User-generated content, event ingestion, and API payload size.
  • Download traffic: API responses, media delivery, and CDN bandwidth.
  • CDN considerations: Use edge caching for static content and heavy media to reduce origin load and improve user experience.
Capacity estimation is one of the biggest differentiators in senior-level interviews.

BAB C — API Design Fundamentals

C.1 REST API Design

Good API design sets the foundation for clear system boundaries. Use RESTful conventions to make interfaces predictable and easy to consume.

  • Resource-oriented design: Model API endpoints around nouns like /users, /posts, /videos, and /links.
  • URL conventions: Keep paths consistent and hierarchical, using plural nouns and nested resources where appropriate.
  • HTTP methods: Use GET for reads, POST for creation, PUT/PATCH for updates, and DELETE for removals.

C.2 User Operations

Designing the core CRUD operations first gives you a clear anchor for the rest of the architecture.

  • Create: Define how data enters the system and what validation is required.
  • Update: Choose between full object replacement and partial updates based on API complexity.
  • Delete: Decide whether deletions are hard or soft, and how they impact downstream systems.
  • Retrieve: Optimize read operations with filters, pagination, and projection.

C.3 Designing Scalable APIs

Scalable APIs are not just about the server. They require data-aware design, versioning, and clear controls on the result set.

  • Pagination: Never return unbounded lists; support page or cursor-based pagination.
  • Filtering: Allow clients to limit the data they request so the backend work stays efficient.
  • Sorting: Provide consistent ordering semantics and sensible defaults for results.
  • Versioning: Protect the API contract over time and support backward-compatible iterations.
Every large-scale system starts with well-defined APIs.

BAB D — Design Instagram / Social Media Platform

D.1 Post Creation

Social media post creation is the heart of the product. Design a service that accepts text, images, and videos while validating content, metadata, and permissions.

  • Text posts: Simple content creation with metadata like captions, hashtags, and mentions.
  • Image uploads: Support multi-file upload, image validation, and storage placement.
  • Video uploads: Handle large payloads, background processing, and eventual availability.

D.2 Social Interactions

Likes, comments, and shares are the social signals that make the system sticky. Each interaction has different throughput and consistency characteristics.

  • Likes: High frequency, eventually consistent counters, and cache-friendly read patterns.
  • Comments: Moderated text content with strong read consistency for recent conversations.
  • Shares: Scaled distribution of content through feeds and notifications.

D.3 Following System

The follow graph is one of the most important data structures in a social platform. It drives both visibility and feed generation.

  • Follow users: Represent directed edges and store them in a structure that scales for heavy-read and heavy-write access patterns.
  • Unfollow users: Support fast removal and consistent feed pruning without expensive recomputation.

D.4 News Feed

Generating the news feed is a design choice between precomputing content and computing it on demand.

  • Fan-out on write: Push new content into followers' feeds when posts are created; fast reads but expensive writes.
  • Fan-out on read: Compute the feed when the user requests it; cheaper writes but heavier read latency.
  • Feed generation: Combine chronological ordering, personalization signals, and caching to meet freshness and latency goals.

D.5 Media Processing

Social media systems are media-heavy. Processing pipelines must transform raw uploads into optimized delivery formats and serve them reliably.

  • Object storage: Store originals in a durable object store and serve derived assets from CDN-ready locations.
  • CDN: Cache static assets near users to reduce latency and origin load.
  • Image optimization: Generate multiple resolutions and formats for different devices.
  • Video transcoding: Convert uploads into adaptive bitrate streams and store variants for playback.

D.6 Celebrity Account Optimization

High-follower accounts create extreme scale challenges. Your design must handle superstar traffic without slowing down regular users.

  • High-follower accounts: Use fan-out optimizations, partial materialization, and selective caching for superstar feeds.
  • Feed scalability challenges: Avoid loading a single feed entry to all followers in one operation; shard and batch pushes where possible.
Classic FAANG system design interview question.

BAB E — Design YouTube / Video Streaming Platform

E.1 Upload Workflow

Video upload is the entry point for a streaming platform. Build a reliable ingestion pipeline that accepts large chunks, validates metadata, and persists content.

  • Video ingestion: Support resumable uploads, chunked transfer, and user feedback on progress.
  • Metadata management: Store titles, descriptions, tags, categories, and access controls separately from raw media.

E.2 Processing Pipeline

Transcoding and content processing are the backbone of a video platform. This pipeline turns uploads into playable formats and extracts search-friendly metadata.

  • Video transcoding: Generate multiple resolutions and bitrates for adaptive playback.
  • Content processing workflow: Run thumbnail generation, moderation checks, and metadata enrichment as separate pipeline stages.

E.3 Streaming Architecture

Streaming large video files efficiently requires edge delivery and adaptive streaming techniques.

  • HLS encoding: Segment video into chunks that can be downloaded progressively by players.
  • Adaptive bitrate streaming: Serve different quality levels based on network conditions to avoid buffering.
  • CDN integration: Push playback segments to the edge to scale global delivery and reduce origin bandwidth.

E.4 Storage Strategy

Video storage is expensive, so a practical design uses multiple tiers and caches to balance cost and performance.

  • Video storage: Keep master copies in durable object storage and hot playback segments in faster edge caches.
  • Metadata storage: Use a low-latency datastore for search and recommendation, separate from raw media storage.
Covers media-heavy distributed systems.

BAB F — Design TinyURL

F.1 URL Shortening Fundamentals

URL shortening appears simple, but it demands durability, correctness, and extreme read performance at scale. The system must convert long URLs into compact aliases and redirect without failure.

  • Long URL to short URL: Store the original URL and return a short code that can be resolved quickly.
  • URL redirection: Redirect requests to the original destination with minimal latency and high availability.

F.2 Short URL Generation

Generating unique short URLs is the key design question. Choose a strategy that minimizes collisions and supports scale.

  • Random strings: Generate short codes with enough entropy to avoid collisions, and retry on duplicates.
  • Hash-based approach: Use a hash function on the long URL and encode the result, while handling collisions carefully.
  • Counter-based approach: Use a sequential ID and encode it into a compact form; this is simple and easy to shard.

F.3 Base62 Encoding

Base62 encoding compresses numeric IDs into short, URL-safe strings. It's the most common technique for building memorable codes.

  • ID generation: Convert a numeric sequence into a compact code using digits, lowercase, and uppercase letters.
  • Encoding and decoding: Ensure the mapping is reversible so the system can efficiently resolve the short code back to the original URL.

F.4 Collision Handling

Any generation strategy must be resilient to collisions. Plan for detection, retries, and consistency in your storage layer.

  • Duplicate detection: Use unique constraints or idempotent checks to catch collisions at write time.
  • Unique generation strategies: Implement fallback mechanisms like salted hashing or alternate counters when a collision is detected.

F.5 Scalability Concerns

A URL shortener is mostly a read-heavy system. Design for millions of redirects per second while keeping the write path lightweight and fast.

  • High traffic redirection: Use caching and distributed key-value storage for the lookup path.
  • Cache optimization: Cache the most popular short URLs at the edge and ensure cache invalidation is simple.
One of the most frequently asked interview questions.

BAB G — Design Rate Limiter

G.1 Why Rate Limiting Matters

Rate limiting protects services from abuse, helps maintain stability, and enforces fair use. In interviews, rate limiting is a great way to show that you can think beyond feature correctness to system resilience.

  • Abuse prevention: Prevent malicious or accidental overuse that could degrade service quality.
  • Resource protection: Guard downstream systems and preserve capacity for legitimate users.

G.2 Placement of Rate Limiter

Where you place the rate limiter changes the design entirely. You can apply limits at the edge, gateway, or service layer depending on the level of control you need.

  • API Gateway: Ideal when you need centralized enforcement and a consistent policy across many services.
  • Service Layer: Use per-service limits when each backend needs its own throttling rules or when latency is critical.

G.3 Storage Design

Rate limiter state must be fast and durable enough to support the counting algorithm. Choose the right datastore for your expected traffic and consistency needs.

  • Counter storage: Use in-memory stores or low-latency databases for frequent increments and reads.
  • Rule storage: Store rate limit policies separately so they can be updated without redeploying the enforcement component.

G.4 Algorithms

Different algorithms trade precision for performance. Interviewers expect you to know the most common ones and when each is appropriate.

Token Bucket

The token bucket algorithm allows bursts while enforcing an average rate. It's a strong choice for APIs that can tolerate short spikes.

  • Token generation: Refill tokens at a steady rate and allow requests to consume tokens.
  • Burst handling: Let traffic burst until the bucket is empty, then throttle new requests.

Leaky Bucket

The leaky bucket enforces a fixed output rate regardless of input bursts. It is simple and predictable, making it a good fit for smoothing traffic to downstream services.

  • Fixed output rate: Drain requests at a constant speed, buffering excess requests until the system can handle them.

Fixed Window Counter

The fixed window counter is the simplest algorithm. It works in fixed intervals, but may allow bursts at window boundaries.

  • Simplicity trade-offs: Easy to implement, but can miscount requests near the edge of windows.

Sliding Window Log

The sliding window log is the most accurate algorithm, tracking timestamps for each request. It is heavier on storage but provides precise enforcement.

  • Accurate request counting: Record each request timestamp and count only those within the active window.

Sliding Window Counter

The sliding window counter is a hybrid that balances accuracy and performance. It uses multiple sub-windows to approximate a sliding view without logging every request.

  • Hybrid solution: Keep counters for recent intervals and interpolate between them for a near-real-time rate estimate.

G.5 Race Condition Handling

Rate limiting is a concurrent system by nature. When many requests increment shared counters at the same time, you need atomic operations and distributed coordination to avoid inconsistencies.

  • Atomic operations: Use atomic increments or compare-and-set operations so counters remain correct even under high concurrency.
  • Distributed counters: Keep the counter state in a highly available store that supports fast increments and strong ordering where needed.
  • Sharded counters: Split counters across shards to reduce contention and then merge totals safely for analytics or enforcement.
Rate limiter correctness depends on both the algorithm and the way shared state is updated under concurrent load.

BAB H — Design WhatsApp / Chat System

H.1 One-to-One Messaging

One-to-one chat should deliver messages quickly and handle users who are offline without losing data.

  • Message delivery: Ensure reliable sends and retries until the recipient confirms receipt.
  • Offline users: Store messages until the recipient reconnects and then deliver them in order.

H.2 Group Messaging

Group chats need a scalable architecture for fan-out and member notifications so messages reach everyone without overwhelming the system.

  • Group architecture: Track membership and group metadata separately from message content.
  • Message fan-out: Push new messages to each member or use a pull model depending on scale and latency needs.

H.3 Online Presence

Online status tells users whether people are available, but it also creates very high-frequency state updates.

  • Last seen: Store lightweight timestamps and update them only when necessary to avoid excessive churn.
  • Online status: Use heartbeat signals and presence services to track active connections.

H.4 Message Status

Message states like sent, delivered, and read create a richer experience, but they also introduce more distributed state to manage.

  • Sent: Confirm that the message left the sender's device and entered the system.
  • Delivered: Confirm that the message reached the recipient's device or server.
  • Read: Track the final acknowledgement when the recipient opens the message.

H.5 Media Messaging

Chat systems must support attachments without sacrificing performance or storage efficiency.

  • Images: Upload, compress, and store images separately from text messages.
  • Videos: Handle large files with resumable uploads and separate streaming delivery.
  • Documents: Securely store and share file attachments with proper access control.

H.6 WebSocket Architecture

Real-time chat relies on persistent connections, and handling those connections at scale is one of the hardest parts.

  • Persistent connections: Keep sockets open for low-latency delivery and use heartbeats to detect disconnects.
  • Connection management: Route users to the right server and transfer state cleanly when they reconnect.

H.7 End-to-End Encryption

Encryption keeps messages private even when the transport layer or storage systems are untrusted.

  • Message privacy: Encrypt content on the client before it leaves the device.
  • Encryption workflow: Manage keys and trust so only authorized participants can decrypt messages.
Real-time chat systems combine networking, state, and privacy concerns into one complex design problem.

BAB I — Design Search Engine

I.1 Search Requirements

Start with what the search must do. Keyword search and fuzzy matching are the most common user expectations.

  • Keyword search: Find documents containing the requested terms quickly.
  • Fuzzy search: Handle typos and near matches so users still get relevant results.

I.2 Indexing System

An inverted index is the foundation of search. Build it so documents can be found by term rather than by scanning the full dataset.

  • Inverted index: Map terms to the documents that contain them for fast lookups.
  • Search indexing pipeline: Process new content, update the index, and keep it fresh without blocking queries.

I.3 Ranking System

Ranking decides which results users see first. Good ranking balances relevance, popularity, and freshness.

  • Relevance scoring: Score results based on query match quality and term importance.
  • Ranking algorithms: Combine multiple signals to order results meaningfully.

I.4 Query Correction

Search must help users recover from mistakes. Spelling correction and suggestions keep the experience smooth.

  • Spell correction: Detect likely typos and propose corrected queries.
  • Levenshtein distance: Use edit distance to compare query terms with known words.

I.5 Elasticsearch Fundamentals

Modern search often runs on clustered search engines that handle indexing, querying, and scaling.

  • Search cluster: Use separate nodes for indexing, query serving, and coordination.
  • Query execution: Distribute queries across shards and merge results efficiently.
Search architecture is a great way to show how systems can handle large data and low-latency queries.

BAB J — Design Airbnb

J.1 Property Management

Property data is the core of a marketplace. Design create and update flows that keep listings accurate and searchable.

  • Create property: Collect location, pricing, photos, and availability in a consistent model.
  • Update property: Allow hosts to change details without breaking existing bookings or search results.

J.2 Search Properties

Search is the main entry point for guests. Location and availability filters are essential for good booking experiences.

  • Location search: Support geographic filtering and relevance based on the guest's query.
  • Availability search: Show only properties that can actually be booked for the requested dates.

J.3 Booking Workflow

Booking requires a reliable workflow that reserves inventory and handles payments safely.

  • Reservation process: Lock the property for the requested time while the guest completes checkout.
  • Payment flow: Integrate payment processing and confirm the booking only after payment clears.

J.4 Concurrent Booking Problem

Multiple guests may try to book the same property at once. Avoid double bookings with careful concurrency control.

  • Double booking prevention: Use transactional reservation checks or optimistic locking to ensure availability.
  • Distributed locking: Coordinate across services when inventory updates happen in multiple places.
Airbnb is a strong interview example because it combines search, booking workflow, and consistency challenges.

BAB K — Design Notification System

K.1 Notification Types

Notification systems must support different channels and delivery guarantees depending on the message type.

  • Push notifications: Real-time mobile alerts for urgent updates.
  • Email notifications: Reliable delivery for more detailed or delayed messages.
  • SMS notifications: Short text alerts with high delivery reliability.

K.2 User Preferences

User controls are essential to avoid notification fatigue and stay compliant with privacy expectations.

  • Opt-in: Let users choose the notifications they want to receive.
  • Opt-out: Respect users who want to stop certain alerts or channels.

K.3 Notification Prioritization

Not all notifications are equal. Separate critical system alerts from marketing or informational messages.

  • Critical notifications: Delivery should be fast and reliable.
  • Marketing notifications: Can be batched or delayed to reduce user disruption.

K.4 Rate Limiting Notifications

Notification streams can overwhelm users and systems if left unchecked, so throttling is important.

  • Prevent notification spam: Enforce limits per user and per channel.

K.5 Decoupled Architecture

Async, queue-based delivery makes notifications scalable and resilient.

  • Queue-based processing: Buffer notifications and retry failed deliveries.
  • Async delivery: Separate sending from event generation so spikes do not break the core product.
Notifications are a perfect example of event-driven architecture and user-facing reliability.

BAB L — Design Logging Platform

L.1 Log Collection

Collecting logs from many services is the first step in building observability for a distributed system.

  • Service logs: Gather application and infrastructure logs from every component.
  • Centralized logging: Send logs to a common system where they can be searched and analyzed.

L.2 Kafka Pipeline

A message bus like Kafka helps buffer logs and stream them reliably to downstream processors.

  • Buffering events: Decouple producers and consumers so log spikes do not overwhelm the system.
  • Event streaming: Stream logs to analytics and alerting pipelines in real time.

L.3 Flink Processing

Real-time processing lets you aggregate and enrich logs as they arrive.

  • Real-time processing: Compute metrics and alerts from streaming log data.
  • Aggregations: Summarize events for dashboards and trend analysis.

L.4 Storage Layer

Log data is often stored in both hot and cold tiers based on how quickly it needs to be accessed.

  • Hot storage: Keep recent logs in fast storage for debugging and monitoring.
  • Cold storage: Archive older logs cost-effectively for compliance and long-term analysis.

L.5 Query & Dashboard

The final layer is user access: search, monitor, and draw insights from logs.

  • Monitoring: Build dashboards and alerts for important system behavior.
  • Analytics: Use search and aggregation to answer operational questions.
A logging platform is a great case study for combining streaming, storage, and observability.

BAB M — Design OTP Service

M.1 OTP Generation

OTP systems need secure codes that are easy to generate and hard to guess.

  • Secure code generation: Use strong randomness and avoid predictable sequences.

M.2 OTP Validation

Validation must be fast and ensure codes cannot be reused after they expire.

  • Verification workflow: Check the provided code, expire it after use, and reject invalid attempts.

M.3 Cache Integration

Redis or similar caches make OTP validation fast while keeping the system stateless.

  • Redis usage: Store OTP values and expiration times in a low-latency cache.

M.4 Security Measures

Protect OTP flows from abuse with expiration, rate limiting, and replay prevention.

  • Expiration: Keep OTP codes valid only for a short window.
  • Rate limiting: Prevent repeated generation and validation attempts from the same user or device.
  • Replay prevention: Ensure an OTP cannot be reused after it has been accepted.

M.5 High Availability

OTP services are often part of authentication, so they must remain available across regions.

  • Multi-region support: Replicate OTP state or use regional caches to serve users near them.
  • Failover strategies: Keep fallback paths so authentication still works during outages.
OTP systems are small but critical, and they highlight security plus reliability requirements.

BAB N — Core Infrastructure Concepts

N.1 Scaling

Understand the difference between growing vertically and horizontally so you choose the right scaling strategy.

  • Vertical scaling: Add more power to a single machine for easier management but limited capacity.
  • Horizontal scaling: Add more machines to increase capacity and resilience.

N.2 Load Balancers

Load balancers distribute traffic and can also provide health checks, SSL termination, and routing intelligence.

  • L4 load balancer: Works at the transport layer for simple, fast traffic distribution.
  • L7 load balancer: Inspects HTTP requests and routes them based on headers, paths, and content.

N.3 Database Scaling

Scaling databases requires replication for reads and sharding for writes.

  • Replication: Copy data across nodes for availability and read scalability.
  • Sharding: Split data into partitions so writes and storage scale horizontally.

N.4 Caching

Caching speeds up systems by keeping hot data closer to the application and avoiding repeated backend work.

  • Redis: A versatile in-memory store for low-latency caching.
  • Memcached: A lightweight cache for simple key-value access patterns.

N.5 CDN

Content delivery networks make global traffic faster by serving static assets close to users.

  • Static asset delivery: Cache images, scripts, and media at the edge.
  • Global distribution: Use multiple edge locations to reduce latency worldwide.

N.6 Messaging Systems

Messaging systems decouple services and enable asynchronous processing.

  • Kafka: Durable event streaming for high-throughput data flows.
  • RabbitMQ: Flexible queuing for task distribution and event delivery.

N.7 API Gateway

API gateways centralize routing, authentication, and cross-cutting concerns like rate limiting.

  • Routing: Send requests to the correct backend service.
  • Authentication: Enforce access controls before traffic reaches application services.
  • Rate limiting: Apply policies centrally for consistent protection.
Infrastructure concepts are the reusable building blocks behind every large system design.

BAB O — Networking Fundamentals

O.1 DNS

DNS translates human-friendly names into IP addresses so users can reach your service without memorizing endpoints.

  • Name resolution: Map domain names to server addresses reliably and quickly.

O.2 TCP & UDP

Choose the right transport protocol depending on whether you need reliability or speed.

  • Reliability: TCP provides ordered, reliable delivery.
  • Performance trade-offs: UDP can be faster for real-time traffic but needs handling for packet loss.

O.3 HTTP

HTTP is the core protocol for web APIs and browser traffic. Understand its request-response model and semantics.

  • Request-response model: Clients send requests and servers return status codes and payloads.

O.4 WebSocket

WebSocket provides persistent bidirectional communication for real-time features like chat and live updates.

  • Persistent communication: Keep a connection open for fast, duplex message exchange.

O.5 Reverse Proxy

Reverse proxies route traffic, provide security, and help manage load across backend servers.

  • Nginx: A common reverse proxy for traffic routing and SSL termination.
  • Traffic routing: Direct requests to the right service based on host, path, or headers.
Networking fundamentals are essential for designing systems that actually work in production.

BAB P — Advanced Backend Technologies

P.1 GraphQL

GraphQL offers flexible querying and lets clients request exactly the data they need.

  • Flexible querying: Clients shape responses, which can reduce over-fetching and simplify API usage.

P.2 gRPC

gRPC is a high-performance RPC framework useful for service-to-service communication in microservices environments.

  • High-performance RPC: Use binary protocols and generated client code for fast internal APIs.

P.3 REST APIs

REST remains the standard for web APIs and is often the best choice for external-facing services.

  • Standard web APIs: Use HTTP and resource-based patterns for broad compatibility and simplicity.

P.4 Message Queues

Message queues provide reliable async communication and help absorb bursts in traffic.

  • Async communication: Decouple producers and consumers so services can scale independently.
Modern backend systems often combine GraphQL, gRPC, REST, and queues depending on use case.

BAB Q — Data Storage & Caching

Q.1 SQL Databases

Relational databases are ideal for structured data and strong consistency requirements.

  • Relational modeling: Design normalized schemas that represent entities and relationships clearly.

Q.2 NoSQL Databases

NoSQL stores are useful when your data model is flexible or you need extreme scale.

  • Key-value: Fast access for simple lookup patterns.
  • Document stores: Flexible structures for evolving application data.

Q.3 Object Storage

Object storage is the right choice for large unstructured files like media and backups.

  • S3-style systems: Durable, scalable storage with URLs and metadata support.

Q.4 Caching Strategies

Good caching reduces latency and backend load when used with the right invalidation strategy.

  • Cache Aside: Load data into cache on demand.
  • Write Through: Update cache and datastore together.
  • Write Back: Write to cache first, then persist later.
  • Read Through: Let the cache load missing values automatically.
Storage and caching decisions are the highest ROI topics in system design interviews.

BAB R — Advanced Distributed Systems

R.1 Cloud Computing

Cloud-native design uses managed services and elastic infrastructure to build systems that scale on demand.

  • Cloud-native systems: Use services like compute, storage, and networking that scale automatically.

R.2 Logging & Monitoring

Observability is central to distributed systems. Track logs, metrics, and traces so you can diagnose failures quickly.

  • Observability: Collect the right signals to understand system health and behavior.

R.3 Consistent Hashing

Consistent hashing helps distribute data and traffic evenly while minimizing rebalancing when nodes change.

  • Distributed partitioning: Assign keys to nodes so the system can scale without massive reshuffling.

R.4 CAP Theorem

The CAP theorem explains the trade-offs between consistency, availability, and partition tolerance in distributed systems.

  • Consistency: Every read sees the latest write.
  • Availability: The system responds even when some nodes fail.
  • Partition tolerance: The system continues despite network splits.

R.5 Database Indexing

Indexing speeds queries by organizing data for fast lookups, which is essential for large-scale datasets.

  • B-Tree concepts: Use balanced tree structures to index relational data efficiently.
  • Query optimization: Choose indexes that match common query patterns.

R.6 Kafka Event Streaming

Kafka is a popular event-streaming platform for building resilient, event-driven systems.

  • Event-driven systems: Use streams to decouple producers from consumers and build scalable pipelines.
Advanced distributed systems are about making the right trade-offs at scale.
