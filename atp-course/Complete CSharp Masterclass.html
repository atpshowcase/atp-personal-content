BAB A. Introduction & Fundamentals

A.1 Introduction to C#

C# (pronounced "C Sharp") is a modern, object-oriented, and type-safe programming language developed by Microsoft as part of the .NET initiative. It is designed to be simple yet powerful, enabling developers to build a vast range of applications on the .NET ecosystem.

Analogy: C# is like a "Swiss Army Knife" for high-tech engineers. It's sturdy, incredibly well-designed, and has a dedicated tool for almost every job you'll face—whether you're carving a wooden toy (simple script) or building a skyscraper (enterprise system).
  • C# vs. .NET: C# is the language you write, while .NET is the "engine" (runtime and libraries) that runs it.
  • Versatility: Used for Web (ASP.NET), Desktop (WPF/WinForms), Mobile (MAUI), and Games (Unity).
  • Modern Features: Includes Garbage Collection (automatic memory management), strong typing, and async/await support.
  • Setup: Most developers use "Visual Studio" (heavyweight) or "VS Code" (lightweight) combined with the .NET SDK.

A.2 First Program & Basics

Every journey starts with a single line of code. In C#, even a simple "Hello World" requires a bit of structure, which helps keep larger programs organized as they grow.

Analogy: Writing your first program is like learning to say "Hello" in a new language. It's simple, but it represents the first successful bridge between your mind and the machine's logic.
using System; // Import the System namespace

namespace MyFirstApp {
  class Program {
    // The Main method is the starting point of every C# program
    static void Main(string[] args) {
      Console.WriteLine("Hello, C# World!");
    }
  }
}
  • Namespaces: Used to organize code and prevent naming conflicts.
  • Classes: The basic building block of C# applications (everything lives in a class).
  • Main Method: Where the execution begins.

A.3 Variables & Data Types

Variables are named storage locations in memory. C# is a **strongly-typed** language, meaning you must be explicit (or let the compiler infer) what kind of data you are storing to prevent errors and optimize memory usage.

Analogy: Data types are like different containers in a kitchen. You don't put soup in a colander, and you don't store flour in a drinking glass—choosing the right container (type) prevents a mess (bugs).

Integral (Whole Number) Types

  • byte/sbyte: 8-bit integers (Range: 0 to 255 or -128 to 127). Used for raw data or very small counters.
  • short/ushort: 16-bit integers. Useful for saving memory in large arrays of small numbers.
  • int/uint: 32-bit integers. The standard choice for whole numbers (up to 2.1 billion).
  • long/ulong: 64-bit integers. Use this for massive numbers (e.g., world population or high-precision timestamps).

Floating-Point (Decimal) Types

  • float: 32-bit (7 digits precision). Fast, but can have rounding errors. Used in graphics/physics.
  • double: 64-bit (15-17 digits precision). The default for general decimal math.
  • decimal: 128-bit (28-29 digits precision). **Crucial for Financial Apps** because it eliminates rounding errors found in float/double.

Text, Character & Logical Types

  • char: A single 16-bit Unicode character (e.g., 'A'). Must use single quotes.
  • string: A sequence of Unicode characters (e.g., "Hello World"). Must use double quotes.
  • bool: Can only be true or false. Essential for control flow logic.

Advanced Type Concepts

  • var (Type Inference): You write var name = "Alice"; and the compiler *figures out* it's a string. It's still strongly typed but makes code cleaner.
  • nullable (int?): Allows a value type to hold a null value, useful for database data.

Special & Omnipresent Types

  • object: The "Parent of All Types." Every single type in C# (even `int`) inherits from object. You can store anything here, but you'll need to cast it back later.
  • dynamic: Tells the compiler to skip type checking until the program is actually running. Powerful for COM interop or JSON parsing, but dangerous if not used carefully.
  • void: Used only as a return type for methods to indicate that the method **returns no value**.

Memory Architecture: Stack vs. Heap

  • Value Types (Stack): `int`, `char`, `bool`, `struct`. Stored in a fast, small part of memory. When they leave a method, they are cleared instantly.
  • Reference Types (Heap): `string`, `class`, `array`. Stored in a larger, dynamic part of memory. Only a "pointer" (address) stays on the stack. They are cleaned up by the **Garbage Collector (GC)**.

A.4 Operators & Expressions

Operators are symbols used to perform calculations, compare values, or execute logical operations. They are the "verbs" that allow your program to actually *do* things with data.

Analogy: Operators are like the basic controls on a calculator. Plus, minus, bigger than, and "and/or" logic allow you to process raw numbers into meaningful results.
  • Arithmetic: + (add), - (sub), * (mul), / (div), % (modulus/remainder).
  • Relational: == (equal), != (not equal), >, <, >=, <=.
  • Logical: && (AND), || (OR), ! (NOT).
  • Assignment: =, +=, -= (shorthand for updating variables).

A.5 Debugging Basics

Bugs are a natural part of coding. Professional developers don't write "perfect" code on the first try; they use powerful tools to find and fix errors systematically.

Analogy: Debugging with breakpoints is like having a "Pause Button" for time. You can stop the world, look at exactly what's in every variable, and see exactly where things went sideways before the crash.
  • Syntax Errors: Typos or grammar mistakes in code (highlighted in red, program won't run).
  • Runtime Errors: Logic mistakes that happen while the app is running (e.g., dividing by zero).
  • Breakpoints: Red dots you place to "freeze" the code at a specific line.
  • Stepping: Moving through code line-by-line using "Step Over" or "Step Into".

BAB B. Control Flow

B.1 Conditional Statements

Control flow allows your program to make decisions. In C#, we use `if/else`, `else if`, and the `switch` statement to branch logic based on specific conditions.

Analogy: Conditional statements are like a "Choose Your Own Adventure" book. If you walk through the door, go to page 20; otherwise, go to page 25.

Deciding with IF/ELSE

if (hour < 12) {
  Console.WriteLine("Good Morning!");
} else if (hour < 18) {
  Console.WriteLine("Good Afternoon!");
} else {
  Console.WriteLine("Good Evening!");
}

Using the Switch Statement

When comparing a single variable against many potential values, a `switch` statement is often cleaner than a long chain of `if/else if` blocks.

string day = "Monday";
switch (day) {
  case "Monday":
    Console.WriteLine("Busy day ahead!");
    break;
  case "Friday":
    Console.WriteLine("Weekend is near!");
    break;
  default:
    Console.WriteLine("Just another day.");
    break;
}

Ternary Operator (Shorthand)

You can write a simple if/else in a single line using condition ? truePart : falsePart;.

string status = (age >= 18) ? "Adult" : "Minor";

B.2 Boolean Logic

Combining conditions requires logical operators. Understanding these "gates" is critical for writing precise logic.

Analogy: Boolean logic is like a strict club entrance rule. "You can enter IF you have a ticket AND you are wearing shoes"—if either one is false, you're staying outside.
  • AND (&&): Returns true ONLY if both sides are true.
  • OR (||): Returns true if at least ONE side is true.
  • NOT (!): Reverses the boolean value (true becomes false).

B.3 Loops

Loops allow you to execute a block of code repeatedly. Selecting the right loop depends on whether you know the iteration count beforehand.

Analogy: A loop is like a gym instructor telling you to do 10 pushups. Instead of them saying "Do a pushup" ten times manually, they just say "Do this 10 times."

The "For" Loop (Fixed Count)

Best used when you know exactly how many times you need to repeat.

for (int i = 0; i < 5; i++) {
  Console.WriteLine("Iteration: " + i);
}

The "While" Loop (Condition-Based)

Repeats as long as a condition is true. Be careful of "Infinite Loops" if the condition never becomes false!

while (health > 0) {
  Console.WriteLine("Still Fighting...");
  health -= 10;
}

The "Foreach" Loop (Collection-Based)

The standard way to iterate through lists and arrays in C#. It handles the indexing for you automatically.

string[] names = {"Alice", "Bob", "Charlie"};
foreach (string name in names) {
  Console.WriteLine("Hello, " + name);
}

B.4 Loop Control

Keywords that modify the normal flow of a loop execution.

Analogy: break is like an E-stop on a treadmill—you stop everything immediately. continue is like a runner jumping over a single puddle—you skip one step and keep going.
for (int i = 1; i <= 10; i++) {
  if (i == 3) continue; // Skip number 3
  if (i == 7) break;    // Stop the loop completely at 7
  Console.WriteLine(i); 
}
// Output: 1, 2, 4, 5, 6

BAB C. Methods & Functions

C.1 Methods Basics

Methods are blocks of code that perform a specific task and can be "called" whenever needed. They promote the **DRY (Don't Repeat Yourself)** principle by grouping logic into reusable units.

Analogy: A method is like a "Kitchen Appliance." You put in raw ingredients (Parameters), press a button (Execution), and it gives you back a finished dish (Return value). You don't need to know how the gears turn, just how to use the interface.

Structure of a Method

  • Return Type: What the method sends back (e.g., int, string, or void if it returns nothing).
  • Parameters: The values you "pass into" the method so it can do its job.
  • Body: The actual code that runs inside the curly braces {}.
// A method that returns a value
int Add(int a, int b) {
  return a + b;
}

// A method that just performs an action (void)
void Greet(string name) {
  Console.WriteLine("Hello, " + name);
}

C.2 Advanced Methods

C# offers powerful ways to make methods more flexible, allowing them to handle different data types or return multiple values.

Analogy: Overloading is like a coffee machine with different sized cups. Whether you want a Small, Medium, or Large, the button says "Coffee," but the machine knows exactly how much water and beans to use for each based on the "input" (the cup size).

Method Overloading

Defining multiple methods with the same name but different parameters (different "signatures").

int Multiply(int a, int b) => a * b;
double Multiply(double a, double b) => a * b; // Overloaded!

Ref & Out Keywords

  • ref: Passes a variable by reference. The value must be initialized before passing.
  • out: Used to return multiple values from a single method. The variable must be assigned a value *inside* the method.

Optional Parameters

You can provide default values for parameters, making them optional when the method is called.

void Log(string message, bool isError = false) { /* logic */ }
Log("Success"); // isError is false by default
Log("Failed", true); // explicitly set to true

C.3 Scope & Variables

Scope determines where a variable is "visible" and accessible. In C#, scope is defined by the level of indentation and curly braces {}. Understanding scope prevents variable name collisions and unintended side effects.

Analogy: Scope is like a "Private Conversation." If you whisper a secret inside a small room (Local scope), people in the hallway (Global/Class scope) can't hear you. However, you can hear someone shouting from the hallway while you are in the room.

Scope Types in C#

  • Local Scope: Variables declared inside a method. They are "Short-lived" and disappear when the method finishes.
  • Block Scope: Variables declared within an if, for, or while block. They only exist inside those specific braces.
  • Class Level (Fields): Variables declared inside the class but outside any method. They are "Long-lived" and accessible to all methods in the class.

Example: Visualizing Scope & Shadowing

class Player {
  // Class Level Scope (Field)
  int score = 100; 

  void UpdateScore(int score) { // Local Scope (Parameter)
    // SHADOWING: The local 'score' hides the class field 'score'
    Console.WriteLine(score); // Prints the local parameter value
    
    // Use 'this' keyword to access the class level field
    Console.WriteLine(this.score); // Prints 100
    
    if (true) {
        int bonus = 50; // BLOCK SCOPE
        Console.WriteLine(bonus); // Works here
    }
    // Console.WriteLine(bonus); // ERROR: bonus doesn't exist here!
  }
}

Pro Tip: To avoid confusion, C# naming conventions suggest using an underscore for private fields (e.g., _score) to clearly distinguish them from local variables.

BAB D. Data Structures

D.1 Arrays

Arrays are fixed-size collections that store multiple elements of the same type in contiguous memory. Once created, their size cannot be changed.

Analogy: An array is like a physical egg carton. It has a fixed number of slots, and once you buy a 12-slot carton, you can't magically make it fit 13 eggs—you need an entirely new, bigger carton.

Working with Arrays

// Declaration and Initialization
int[] numbers = new int[5]; // Empty array of size 5
string[] fruits = {"Apple", "Banana", "Cherry"}; // Initialized array

// Accessing elements (Zero-based indexing)
Console.WriteLine(fruits[0]); // Output: Apple

// Multi-dimensional Arrays
int[,] matrix = new int[3, 3]; // 3x3 Grid

D.2 Collections

C# provides the System.Collections.Generic namespace, which contains dynamic structures that can grow or shrink as your program runs.

Analogy: A List is like a digital shopping list on your phone. You can add items as you think of them and delete them once they're in the cart—it grows and shrinks with your needs, unlike a piece of paper with fixed space.
  • List<T>: The most common collection. Ideal for ordered items that change frequently. (e.g., List<string> users = new List<string>();).
  • Dictionary<K, V>: Stores data in key-value pairs. Optimized for lightning-fast lookups using the Key.
  • Legacy Collections: Avoid ArrayList and Hashtable in modern C# development; they are not type-safe and cause performance issues due to "Boxing/Unboxing."
List<int> scores = new List<int> { 90, 85, 100 };
scores.Add(95); // Dynamic growth!

Dictionary<int, string> studentIdMap = new Dictionary<int, string>();
studentIdMap.Add(101, "John Doe");

D.3 LINQ Basics

LINQ (Language Integrated Query) allows you to process and filter data from any collection using a expressive, readable syntax.

Analogy: LINQ is like a "Baggage Scanner" at an airport. You don't open every bag manually to check its weight; you just set a filter (e.g., "Find all bags over 30kg") and the scanner gives you the results instantly.

Common LINQ Methods

  • Where: Filters data based on a condition.
  • Select: Transforms each element (e.g., getting just the names from objects).
  • OrderBy / OrderByDescending: Sorts the results.
using System.Linq;

var highScores = scores.Where(s => s > 90).OrderByDescending(s => s).ToList();
// Finds scores over 90 and sorts them from highest to lowest.

BAB E. Object-Oriented Programming (OOP)

E.1 Classes & Objects

C# is an OOP language at its core. Classes are blueprints that define structure and behavior, while Objects are the memory-allocated instances of those blueprints.

Analogy: A Class is a blueprint for a car. It says a car has wheels and can drive. An Object is the specific red car sitting in your driveway right now.
class Player {
  public string Name { get; set; } // Property
  public int Health { get; private set; } // Encapsulated property

  // Constructor: Initializes the object
  public Player(string name) {
    Name = name;
    Health = 100;
  }
}

Player hero = new Player("Arthur"); // Instantiating an object

E.2 Encapsulation

Encapsulation is the core of "Information Hiding." It involves wrapping data (Fields) and code (Methods) into a single unit and strictly controlling access to that unit. This ensures **Data Integrity**—meaning your object's internal state cannot be corrupted by outside code.

Analogy: Encapsulation is like the "Vending Machine" interface. You press a button (Public method), and it gives you a soda. You can't reach in and grab the money or gears (Private data)—the machine protects itself through a strict, predefined interface.

Access Modifier Comparison

If you're still confused by the differences, use the House Analogy to see who is allowed access to various areas:

  • public (The Street): Anyone in the world can walk in. Accessible from any project in your solution.
  • private (Your Bedroom): Only YOU (the class creator) can enter. Accessible only within the same class.
  • protected (The Living Room): Only you and your children (Inheritance) can enter. Accessible to the class and its derived classes.
  • internal (Gated Community): Only neighbors in your assembly/project can enter. Accessible to anyone in the same project.

Technical Implementation

class BaseUnit {
    public int health = 100;      // Accessible to everyone (The Street)
    private int _id = 007;        // Only in BaseUnit (Your Bedroom)
    protected string team = "A";  // To BaseUnit and Medic (Living Room)
    internal string region = "HK";// To anyone in this project (Gated Community)
}

class Medic : BaseUnit {
    void Heal() {
        health = 120; // OK (Public)
        team = "B";   // OK (Protected)
        // _id = 1;   // ERROR: Private to BaseUnit!
    }
}

Why Use Properties Over Fields?

In C#, we keep our variables private (Fields) and expose them through public wrappers (Properties). This allows us to intercept the data and apply "Validation Logic" before it's saved.

Example: Protecting a Bank Account

class BankAccount {
  // Private field (The "Money" inside the safe)
  private decimal _balance; 

  // Public property (The "ATM" interface)
  public decimal Balance {
    get { return _balance; } // You can check the balance
    // NO SETTER: Outside code cannot "Force" a new balance directly!
  }

  public void Deposit(decimal amount) {
    if (amount > 0) _balance += amount; // Logic check
  }
}

Goal: By making the setter private (or removing it entirely), we prevent bugs where a developer accidentally sets your bank balance to a negative number.

E.3 Inheritance

Inheritance allows a "Derived" class to reuse and extend the functionality of a "Base" class. It enables the DRY principle across your object hierarchy.

Analogy: Inheritance is like "Family Genes." A child inherits their parents' height (Base behavior), but they can also learn new skills like playing the piano (Extended behavior) that their parents didn't have.
class Animal {
  public void Eat() => Console.WriteLine("Eating...");
}

class Dog : Animal { // Dog inherits from Animal
  public void Bark() => Console.WriteLine("Woof!");
}

E.4 Polymorphism

Polymorphism allows objects of different types to be treated as a common parent type. This is achieved through Method Overriding using virtual and override.

Analogy: Polymorphism is like the word "Play." You can "Play" a guitar, "Play" a movie, or "Play" a sport. The command is the same, but the result changes based on the object you are interacting with.
class Shape {
  public virtual void Draw() => Console.WriteLine("Drawing Shape");
}

class Circle : Shape {
  public override void Draw() => Console.WriteLine("Drawing Circle");
}

E.5 Interfaces

Interfaces define a contract that implementing classes *must* follow. Unlike inheritance, a class can implement multiple interfaces, allowing for highly flexible and decoupled code.

Analogy: An interface is like a "USB Port." As long as you have a USB plug (Implementation), you can connect to any computer (System), regardless of whether you're a mouse, keyboard, or hard drive.
interface IKillable {
  void Die();
}

class Enemy : IKillable {
  public void Die() { /* play death animation */ }
}

BAB F. Advanced C# Concepts

F.1 Error Handling

Professional code must handle failures gracefully. The try-catch-finally block ensures your app doesn't just vanish when an error occurs, providing a safe way to recover or shut down.

Analogy: Error handling is like the "Airbags" in a car. You hope you never need them, but if something goes wrong, they deploy to prevent total destruction and keep the passengers (users) safe.
try {
  int result = 10 / 0; // Will cause exception
} catch (DivideByZeroException ex) {
  // Specific catch for math errors
  Console.WriteLine("Cannot divide by zero!");
} catch (Exception ex) {
  // 'Exception' is the base class; catches everything else
  Console.WriteLine($"Unexpected error: {ex.Message}");
} finally {
  // Always runs (Cleanup code like closing databases)
  Console.WriteLine("Cleanup process complete.");
}

F.2 File Handling

Reading from and writing to files is essential for data persistence. Modern C# uses the using statement or declaration to ensure resources are disposed of immediately after use.

Analogy: File handling is like a "Locker Room." You can put data inside a locker (Write), lock the door (Close stream), and come back a week later to retrieve it (Read).
using System.IO;

string path = "data.txt";
// Modern 'using' declaration: automatically closes file at end of method
using var writer = new StreamWriter(path);
writer.WriteLine("Hello, Persistence!");

// Simple one-liners for quick tasks:
File.WriteAllText("log.txt", "Event Logged");
string content = File.ReadAllText("log.txt");

F.3 Delegates & Events

Delegates are type-safe function pointers, and Events are the backbone of reactive and UI-driven programming. They allow objects to communicate without being tightly coupled.

Analogy: An Event is like a "Fire Alarm." The alarm doesn't know who is in the building; it just broadcasts a signal. Whoever is interested (Subscribers) decides how to react (Run, Call 911, etc.).
  • Action: A built-in delegate for methods that return nothing (void).
  • Func<T>: A delegate that returns a value of type T.
  • Event: A protective wrapper around a delegate that prevents external classes from clearing the subscriber list.

F.4 Generics

Generics allow you to write classes and methods that work with any data type while maintaining full type safety and performance, avoiding "Boxing" (expensive conversions).

Analogy: A Generic is like a "Shipping Container." It doesn't care if you're shipping cars, TVs, or toys—the container handles the transport in exactly the same safe, standardized way.
// Generic Method: 'T' can be anything
public void LogData(T item) {
  Console.WriteLine($"Logging: {item}");
}

LogData(404);      // Works for integers
LogData("Pass");// Works for strings
// Generic Method
public void Display<T>(T item) {
  Console.WriteLine(item);
}

Display<int>(123);
Display<string>("Hello");

BAB G. Multithreading & Async

G.1 Threading Basics

Multithreading is the ability of the CPU to manage several "work threads" at once. Instead of your program running in a single straight line, it can "split" into multiple paths that run at the same time. This is essential for keeping your apps responsive and taking full advantage of modern multi-core processors.

Analogy: Threading is like a "Busy Kitchen." If you only have one chef, they have to boil the pasta, then wait, then chop the onions, then wait. In a multithreaded kitchen, you have one chef for pasta and one for onions—both work at the same time, and dinner is ready in half the time.
  • Parallelism: Doing two things at the exact same physical moment on different CPU cores.
  • System.Threading: The original .NET namespace used to create and control raw threads manually.
  • Main Thread: The "Boss" thread that starts your app. In UI apps, if this thread gets stuck doing heavy work, the whole window freezes (Not Responding).
  • Worker Threads: Extra "hands" you hire to do heavy lifting in the background without disturbing the boss.

Working with Raw Threads

Even though modern C# uses Tasks (see G.2), understanding raw threads helps you grasp how the OS actually handles concurrency. You create an instance of Thread, give it a method to run, and call Start().

using System;
using System.Threading;

class Program {
    static void Main() {
        // Create a new thread and tell it which method to run
        Thread worker = new Thread(DoHeavyWork);
        worker.Start(); // Starts the background work

        // Meanwhile, the Main thread keeps going
        for (int i = 0; i < 3; i++) {
            Console.WriteLine("Main thread is working...");
            Thread.Sleep(500); 
        }

        worker.Join(); // Wait here until the worker finishes
        Console.WriteLine("All work complete!");
    }

    static void DoHeavyWork() {
        Console.WriteLine("Worker: Starting 3s task...");
        Thread.Sleep(3000); 
        Console.WriteLine("Worker: Task finished!");
    }
}

The Danger: Race Conditions

A Race Condition happens when two threads try to change the same variable at the exact same time. It's like two people trying to write on the same piece of paper at once—the result is corrupted data.

Analogy: Imagine a "Joint Bank Account." If two people withdraw $100 at the exact same millisecond, and the system isn't careful, it might process BOTH withdrawals before updating the balance, leaving you with less money than the bank thinks you have.

To fix this, we use Locking. A lock ensures only one thread can touch a piece of code at a time.

private static readonly object _locker = new object();
private static int _balance = 1000;

static void Withdraw(int amount) {
    lock (_locker) { // Only one thread enters at a time
        if (_balance >= amount) {
            _balance -= amount;
            Console.WriteLine($"Withdrawn: {amount}. New Balance: {_balance}");
        }
    }
}

G.2 Asynchronous Programming (TAP)

The Task-based Asynchronous Pattern (TAP) uses async and await to keep your app "alive" and responsive during long operations like network downloads or database queries.

Analogy: Async is like "Ordering Coffee." You don't stand at the counter staring at the machine until it's done (Blocking). You take a pager (Task), go sit down and continue reading (Non-blocking), and the pager buzzes (Await) when the coffee is ready.

Why Asynchrony Matters?

In desktop and mobile apps, if you run a heavy task "synchronously" on the Main Thread, the entire UI freezes. Users can't click buttons or scroll. Async prevents this freeze.

using System.Net.Http;
using System.Threading.Tasks;

public async Task<string> DownloadWebPageAsync(string url) {
  HttpClient client = new HttpClient();
  
  // The 'await' keyword releases the thread to do other work 
  // until the download is finished.
  string content = await client.GetStringAsync(url);
  
  return content;
}

Key Concept: await is NOT "Wait". It is an instruction to "Pause this method here, free up the thread to handle other UI events, and come back when this Task is done."

BAB H. Working with Databases

H.1 Database Basics

Enterprise applications live and die by their data. Most C# apps use Relational Databases (RDBMS) like SQL Server, PostgreSQL, or SQLite to store structured information.

Analogy: A database is like an "organized warehouse." SQL is the language you use to tell the warehouse robot exactly which box to bring you.

Core Concepts:

  • Tables: Containers for data (e.g., "Users" table).
  • CRUD: The four basic operations: Create, Read, Update, and Delete.
  • Primary Key: A unique ID for every row (like a Social Security number).
  • Relationships: Connecting data between tables (e.g., an Order "belongs to" a User).

H.2 Connecting C# with Database

To talk to a database from C#, you generally have two choices: the low-level "mechanical" way or the high-level "automatic" way. Understanding both is key to building professional applications.

Analogy: Database integration is like "Pipe Fitting." ADO.NET is like building the plumbing manually with individual pipes and wrenches—it takes more work, but you know exactly where every drop of water goes. Entity Framework is like hiring a professional plumbing service—they handle the complexity, and you just turn the tap.

1. ADO.NET: The Manual Transmission

ADO.NET is the foundation. It is incredibly fast and lightweight because it has no "translation layer"—you write the SQL yourself. However, it requires a lot of "Boilerplate" code (repetitive setup for connections, commands, and readers).

using System.Data.SqlClient;

string connString = "Server=myServer;Database=myDB;User Id=myUser;Password=myPassword;";
using (SqlConnection conn = new SqlConnection(connString)) {
    conn.Open();
    string sql = "SELECT Name FROM Students WHERE GPA > 3.5";
    SqlCommand cmd = new SqlCommand(sql, conn);
    
    using (SqlDataReader reader = cmd.ExecuteReader()) {
        while (reader.Read()) {
            Console.WriteLine(reader["Name"]);
        }
    }
}

2. Entity Framework Core (EF Core): The Auto-Pilot

EF Core is an ORM (Object-Relational Mapper). It maps your C# classes directly to database tables. You don't write SQL strings; you write LINQ, and EF Core translates it into SQL for you automatically.

  • Productivity: You focus on C# logic instead of string-based SQL queries.
  • Migrations: Automatically synchronizes your C# class changes to the actual database schema.
  • Type Safety: Catch query errors at compile-time instead of waiting for a crash at runtime.
// 1. Define your data model
public class Product {
  public int Id { get; set; }
  public string Name { get; set; }
  public decimal Price { get; set; }
}

// 2. Setup the Database Context (The Bridge)
public class AppDbContext : DbContext {
  public DbSet<Product> Products { get; set; }
}

// 3. Querying with LINQ (Calculated behind the scenes)
using var db = new AppDbContext();
var expensiveProducts = db.Products
                          .Where(p => p.Price > 100)
                          .OrderBy(p => p.Name)
                          .ToList();

Pro Tip: Most modern enterprise apps use Entity Framework for standard work, and only drop down to ADO.NET (or Dapper) for high-performance "hot paths" where every millisecond counts.

H.3 Popular .NET ORMs

While Entity Framework Core is the official Microsoft choice, the .NET ecosystem has several other powerful ORMs that excel in different scenarios. The right choice depends on whether you value Developer Speed or System Performance.

Analogy: Choosing an ORM is like picking a vehicle. EF Core is a luxury SUV—it's comfortable and does almost everything for you. Dapper is a stripped-down Formula 1 car—it's blazing fast, but you have to know how to drive a manual.
  • Dapper (The Micro-ORM): Known as the "King of Performance." Created by the team at Stack Overflow, it performs nearly as fast as raw ADO.NET. It doesn't handle migrations or complex mapping—you write the SQL, and Dapper simply "populates" your C# objects from the results.
  • NHibernate: A mature port of Java's Hibernate. It is incredibly feature-rich and used heavily in legacy enterprise systems that require complex, custom database mapping.
  • SqlSugar / RepoDb: Modern, high-performance alternatives that offer a balance between the simplicity of Dapper and the features of EF Core.
using Dapper;
// Dapper is an "Extension Method" for IDbConnection
// It turns raw SQL results into C# Objects in one line!

string sql = "SELECT * FROM Users WHERE IsActive = @isActive";
var users = connection.Query<User>(sql, new { isActive = true }).ToList();

Core Differences: Use heavy ORMs (EF/NHibernate) when you have complex data relationships, and use micro-ORMs (Dapper/RepoDb) when you need to process millions of records very, very quickly.

BAB I. Desktop Development (WPF)

I.1 Introduction to WPF

Windows Presentation Foundation (WPF) is the modern standard for building Windows desktop apps. It utilizes XAML (an XML-based language) for design and C# for logic.

The MVVM Pattern

Professional WPF development almost always uses the Model-View-ViewModel (MVVM) pattern. This separates the UI (View) from the data (Model) through a middle-man (ViewModel), making the app much easier to test and maintain.

// 1. ViewModel: Prepares data for the View
public class MainViewModel : INotifyPropertyChanged {

        get => _userName;

        }
    }

BAB J. Game Development (Unity)

J.1 Unity Basics

Unity is the world's most popular game engine, and it uses C# as its engine for behavior. Everything in Unity is a GameObject made up of various Components.

Analogy: Unity is like a "Virtual Toybox." You have props (GameObjects) and properties (Components like 'RigidBody' or 'Mesh'), and C# is the "Magic Spell" that brings them all to life.

The LifeCycle

  • Start(): Runs once when the game object is created. Use it for initialization.
  • Update(): Runs once per frame (60+ times per second). Use it for continuous input or movement.
// Unity uses C# Scripts to define behavior
public class PlayerController : MonoBehaviour {
    public float speed = 5.0f;

    void Update() {
        // Move the player based on WASD keys
        float x = Input.GetAxis("Horizontal");
        float z = Input.GetAxis("Vertical");
        transform.Translate(x * speed * Time.deltaTime, 0, z * speed * Time.deltaTime);
    }
}

J.2 C# in Unity

C# scripts in Unity inherit from MonoBehaviour, allowing them to hook into the engine's physics and input systems.

Analogy: Scripting in games is like being a "Director." You write the script (Code), and when you press play, the actors (GameObjects) follow your commands exactly—gravity works, bullets fly, and players win or lose.

BAB K. Clean Code & Architecture

K.1 Clean Code Principles

Write code for humans first, and computers second. Clean code is about clarity, simplicity, and reducing the "cognitive load" for the next developer.

Analogy: Clean code is like a "Well-organized Library." Even if you haven't been there in years, you can find exactly what you need because the labels make sense and everything is in its logical place.
// ❌ Bad Code (Mysterious Names)
void Proc(int d) { foreach(var x in _l) { if(x.Day == d) x.Send(); } }

// ✅ Clean Code (Intent Revealed)
void NotifyUpcomingEvents(int targetDay) {
    var overdueEvents = _eventList.Where(e => e.ScheduledDay == targetDay);
    foreach(var ev in overdueEvents) {
        ev.SendNotification();
    }
}

K.2 SOLID Principles

SOLID is a set of five design principles that make software designs more understandable, flexible, and maintainable.

  • Single Responsibility: A class should have only one reason to change.
  • Dependency Inversion: Your high-level logic should depend on Abstractions (Interfaces), not concrete implementations. This makes swapping a "Real Database" for "Test Data" incredibly easy.

K.3 Code Maintainability

Refactoring is the process of improving internal code structure without changing its external behavior. It's the primary tool for fighting **Technical Debt**.

Analogy: Maintenance is like "Regular Dental Checks." A small cleaning today (Refactoring) prevents an expensive and painful "root canal" (major system failure) next year.
