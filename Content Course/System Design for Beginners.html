CHAPTER A. Introduction to System Design

A.1 What is System Design

System design is the process of designing the architecture of a software system so it can meet both functional and non-functional needs like scalability, reliability, and performance. It’s not about writing code, but about choosing the right components and arranging how they interact.

User Request
    |
    v
[Load Balancer]
    |
    v
[Application Server]
    |
    v
[Database / Cache]
Analogy: system design is like planning a city map — you decide the highways, distribution routes, and facility locations before you start building anything.

A.2 Why It Matters

A badly designed system will be hard to scale, often go down, and be expensive to fix. On the other hand, a good design makes it easier to add new features, helps the team collaborate, and makes the system more resilient to traffic spikes.

  • Sets the system’s limits from the start.
  • Predicts bottlenecks before they happen in production.
  • Gives the team a common language for technical discussions.
  • Foundation for estimating infrastructure costs.
Analogy: like building a strong house foundation. If the foundation is wrong, repainting and renovating the interior is pointless.

A.3 History and Evolution

System design evolved from simple monolithic architectures in the 1990s to microservices, distributed systems, and the cloud-native architecture we know today. Each era brought new challenges: from scalability, to consistency, to observability.

1990s  : Monolith — one server, one database
2000s  : N-tier — separate frontend, backend, database
2010s  : SOA / Microservices — split by domain
2020s  : Cloud-native — serverless, container-based, event-driven
Analogy: like a food stall evolving into a franchise — from a single kitchen to a distributed supply chain system.

CHAPTER B. Requirements

B.1 Functional Requirements

Functional requirements are what the system must be able to do: features, actions, and behaviors visible to users. These usually come from the product team and become the basis for all design decisions.

  • Users can register, log in, and log out.
  • Users can upload and view photos.
  • Users can send and receive real-time messages.
  • Admins can suspend user accounts.
Analogy: this is the menu list at a food stall — what customers can order.

B.2 Non-Functional Requirements

Non-functional requirements are about system quality: how fast, how reliable, how secure. These often determine the architecture chosen. Two systems with the same functional requirements can have very different designs if their NFRs are different.

  • Availability: system must be up 99.9% of the time.
  • Latency: responses within 200ms for 95% of requests.
  • Throughput: handle 10,000 requests per second.
  • Durability: data must not be lost even if the server crashes.
  • Consistency: all users see the same data.
Analogy: this is the food stall’s SLA (Service Level Agreement) — orders must arrive within 10 minutes, food must be hot, and no wrong orders allowed.

B.3 Estimating Scale

Estimating scale is a back-of-the-envelope exercise to predict the volume of data, traffic, and storage the system will need. This is important for choosing the right infrastructure size from the start.

// Example estimate: photo sharing system
DAU (Daily Active Users)   : 10 million
Uploads per user per day   : 2 photos
Average size per photo     : 2 MB

Photos per day             : 10M x 2 = 20 million photos
Storage per day            : 20M x 2MB = 40 TB
Storage per year           : 40TB x 365 = ~14.6 PB

Read/Write ratio           : 100:1 (more viewing than uploading)
Reads per second           : 10M x 10 views / 86400s = ~1,157 RPS
Analogy: before opening a restaurant, first count the tables, kitchen, and daily stock — then choose the location and building size.

CHAPTER C. High-Level Design

C.1 System Overview

High-level design is the big picture of the system: what the main components are and how they’re connected. This doesn’t go into database schema or library choices yet — the focus is on the big blocks.

[Client (Web/Mobile)]
         |
    [CDN / Edge]
         |
  [API Gateway / Load Balancer]
         |
   +-----+-----+
   |           |
[Auth Service] [App Service]
                   |
             +-----+-----+
             |           |
         [Cache]      [Database]
Analogy: this is the building blueprint — you know where the entrance, main room, and emergency exits are before deciding on the finishing details.

C.2 Components and Data Flow

Once the main components are clear, the next step is to define how data flows between them. Requests from the client follow a certain path, and it’s important to understand who reads, who writes, and where bottlenecks can happen.

  • Client — sends requests, receives responses.
  • API Gateway — routing, auth check, rate limiting.
  • Service — business logic and orchestration.
  • Cache — stores frequently read data.
  • Database — main data persistence.
  • Message Queue — async communication between services.
Analogy: like the flow of goods in a warehouse — goods come in through the receiving door, get quality checked, stored on shelves, then shipped to customers.

C.3 Tech Stack Basics

Your tech stack choices affect performance, cost, and team capability. There’s no stack that’s always the best — the best is what fits your scale, budget, and team skills.

// Example stack for an e-commerce startup
Frontend  : React / Next.js
Backend   : Node.js + Express or .NET Core
Database  : PostgreSQL (primary), Redis (cache)
Queue     : RabbitMQ or Kafka
Storage   : AWS S3
CDN       : Cloudflare
Deploy    : Docker + Kubernetes on AWS
Analogy: pick kitchen equipment based on the menu you want to sell — a fried rice stall doesn’t need a professional pizza oven.

CHAPTER D. API Design

D.1 REST Basics

REST (Representational State Transfer) is the most common API architecture style. The main principles: use the right HTTP methods, resources are identified by URLs, and responses must be stateless.

GET    /users/{id}        // get user
      POST   /users             // create new user
      PUT    /users/{id}        // full update
      PATCH  /users/{id}        // partial update
      DELETE /users/{id}        // delete user
Analogy: REST is like ordering at a restaurant — there’s a clear menu, a standard way to order, and predictable responses.

D.2 Endpoints and Structure

Good endpoint design uses nouns, not verbs. URLs should reflect resources, not actions. Use hierarchy to show relationships between resources.

  • Use nouns: /orders, not /getOrders.
  • Use plural: /products, not /product.
  • Nested resource: /orders/{id}/items.
  • Filter via query string: /products?category=shoes&sort=price.
GET  /orders                  // all orders
      GET  /orders/{id}             // single order
      GET  /orders/{id}/items       // items from a specific order
      POST /orders/{id}/cancel      // specific actions can use sub-resources
Analogy: a URL is like a home address — it should be specific and readable by others without a long explanation.

D.3 Versioning and Rate Limiting

Versioning is important so old clients don’t break when the API is updated. Rate limiting protects the system from abuse and unexpected traffic spikes.

// Versioning via URL path (most common)
      GET /v1/products
      GET /v2/products

      // Versioning via header
      Accept: application/vnd.myapp.v2+json

      // Rate limiting response headers
      X-RateLimit-Limit: 1000
      X-RateLimit-Remaining: 942
      X-RateLimit-Reset: 1710644400

      // HTTP status if rate limited
      HTTP 429 Too Many Requests
Analogy: versioning is like book editions — old editions still circulate, new ones have extra features. Rate limiting is like taking turns in a queue — everyone gets served, but not all at once.

BAB E. Data Design

E.1 SQL vs NoSQL

Memilih antara SQL dan NoSQL bukan soal mana yang lebih bagus, tapi soal mana yang sesuai dengan pola akses data dan kebutuhan konsistensi sistem. Keduanya punya trade-off yang berbeda.

SQL (PostgreSQL, MySQL)
+ ACID transactions
+ Relasi antar tabel dengan JOIN
+ Schema strict — cocok data terstruktur
- Sulit di-scale horizontal
- Kurang fleksibel untuk skema yang sering berubah

NoSQL (MongoDB, DynamoDB, Cassandra)
+ Scale horizontal mudah
+ Schema fleksibel (document, key-value, wide-column)
+ Read/write throughput sangat tinggi
- Konsistensi lemah (eventual consistency)
- Tidak cocok untuk complex relational queries
Analogi: SQL itu seperti filing cabinet berlabel rapi — query-nya presisi tapi sulit diperluas. NoSQL itu seperti kotak besar — fleksibel dan cepat buat ambil banyak barang sekaligus.

E.2 Schema Design

Schema yang baik mencegah anomali data, mempercepat query, dan membuat kode aplikasi lebih mudah dimengerti. Normalisasi adalah teknik umum untuk SQL, sedangkan denormalisasi sering dipilih di NoSQL untuk performa baca.

-- Normalisasi (SQL)
CREATE TABLE users (
  id        SERIAL PRIMARY KEY,
  email     VARCHAR(255) UNIQUE NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE orders (
  id        SERIAL PRIMARY KEY,
  user_id   INT REFERENCES users(id),
  total     DECIMAL(10, 2),
  status    VARCHAR(50),
  created_at TIMESTAMP DEFAULT NOW()
);
Analogi: schema itu kayak aturan penyimpanan arsip — kalau berantakan dari awal, makin lama makin susah dicari.

E.3 Indexing and Partitioning

Index mempercepat query read, tapi memperlambat write dan menambah ukuran storage. Partitioning memecah data besar menjadi potongan lebih kecil agar query lebih efisien dan scale lebih mudah.

-- Index pada kolom yang sering di-query
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status);

-- Partitioning (range by date)
CREATE TABLE orders_2026_q1 PARTITION OF orders
  FOR VALUES FROM ('2026-01-01') TO ('2026-04-01');

-- Sharding (horizontal partitioning)
// Shard 1: user_id 1 - 1.000.000
// Shard 2: user_id 1.000.001 - 2.000.000
// Shard 3: user_id 2.000.001 - 3.000.000
Analogi: index seperti daftar isi buku — bikin buku lebih tebal sedikit, tapi bisa langsung lompat ke halaman yang tepat tanpa baca dari awal.

BAB F. Scalability

F.1 Vertical vs Horizontal Scaling

Vertical scaling berarti menambah resource ke satu server (CPU, RAM lebih besar). Horizontal scaling berarti menambah jumlah server. Keduanya punya batas dan biaya berbeda.

Vertical Scaling (Scale Up)
  Server: 4 CPU, 16 GB RAM
  → Upgrade ke: 32 CPU, 256 GB RAM
  + Mudah, tidak perlu ubah kode
  - Ada batas hardware, single point of failure

Horizontal Scaling (Scale Out)
  1 server → 10 server di belakang load balancer
  + Tidak ada batas teoritis
  + Fault tolerant (satu mati, yang lain jalan)
  - Perlu handle state, session, dan consistency
Analogi: vertical scaling kayak beli meja kerja lebih besar. Horizontal scaling kayak tambah meja dan rekrut pegawai baru.

F.2 Load Balancing

Load balancer mendistribusikan traffic ke beberapa server agar tidak ada satu server yang kelebihan beban. Ia juga berfungsi sebagai titik health check dan failover otomatis.

Client
  |
[Load Balancer]   ← algorithm: Round Robin, Least Connections, IP Hash
  |       |       |
[S1]    [S2]    [S3]

// Round Robin: giliran berurutan
// Least Connections: server dengan koneksi aktif paling sedikit
// IP Hash: client yang sama selalu ke server yang sama (sticky session)
Analogi: load balancer itu kayak resepsionis kasir supermarket — ngarahin pembeli ke kasir yang antriannya paling pendek.

F.3 Stateless Systems

Sistem stateless berarti setiap request membawa semua informasi yang dibutuhkan untuk diproses — server tidak menyimpan state sesi di memorinya. Ini memudahkan horizontal scaling karena request bisa dilayani server mana pun.

// Stateful (hindari ini untuk scalable system)
// Server menyimpan session di memori
sessions["user123"] = { cart: [...], token: "abc" }

// Stateless (lebih scalable)
// Session disimpan di Redis / JWT
Authorization: Bearer eyJhbGci...
// Setiap request membawa token → server mana pun bisa proses
Analogi: stateless itu kayak kasir yang tidak perlu ingat wajah pelanggan — semua info ada di struk dan kartu member yang dibawa pelanggan sendiri.

BAB G. Caching

G.1 Purpose of Caching

Cache menyimpan hasil komputasi atau data yang sering diakses di tempat yang lebih cepat diakses — biasanya memory. Tujuannya adalah mengurangi latency, mengurangi beban database, dan meningkatkan throughput sistem secara keseluruhan.

  • Mengurangi query ke database yang berulang.
  • Mempercepat response untuk data yang jarang berubah.
  • Menyerap lonjakan traffic tanpa membebani backend.
Analogi: cache itu kayak buku catatan kecil di meja kerja — daripada ke arsip setiap kali butuh nomor telepon penting, lo tulis di catatan yang gampang dijangkau.

G.2 Cache Strategies

Ada beberapa strategi mengisi dan memperbarui cache. Pemilihan strategi bergantung pada pola baca/tulis data dan toleransi terhadap stale data.

// Cache-Aside (Lazy Loading) — paling umum
function getProduct(id) {
  data = cache.get(id)
  if (data == null) {
    data = db.get(id)
    cache.set(id, data, TTL=300s)
  }
  return data
}

// Write-Through — tulis ke cache dan DB bersamaan
function updateProduct(id, newData) {
  db.update(id, newData)
  cache.set(id, newData)
}

// Write-Behind (Write-Back) — tulis ke cache dulu, DB async
// Lebih cepat, tapi risiko data loss kalau cache crash
Analogi: cache-aside kayak tanya teman dulu sebelum googling. Write-through kayak nulis di papan pengumuman dan buku catatan sekaligus.

G.3 Cache Invalidation

Cache invalidation adalah salah satu masalah paling sulit dalam sistem terdistribusi. Data di cache bisa basi (stale) kalau sumber datanya berubah tanpa cache diupdate. Gunakan TTL dan mekanisme invalidasi yang tepat.

// TTL (Time-To-Live) — cache kadaluarsa otomatis
cache.set("product:123", data, ttl=600)  // expire 10 menit

// Event-based invalidation — hapus cache saat data berubah
on ProductUpdated(productId):
  cache.delete("product:" + productId)

// Cache stampede protection — cegah banyak request serang DB sekaligus
function getWithLock(key):
  data = cache.get(key)
  if data == null:
    if lock.acquire(key, timeout=5s):
      data = db.get(key)
      cache.set(key, data)
      lock.release(key)
  return data
Analogi: cache invalidation kayak kadaluarsa makanan di kulkas — kalau tidak dicek tanggalnya, bisa kasih makanan basi ke tamu.

BAB H. Asynchronous Systems

H.1 Sync vs Async

Komunikasi synchronous berarti pengirim menunggu sampai penerima selesai memproses. Asynchronous berarti pengirim melanjutkan prosesnya sendiri tanpa menunggu. Async cocok untuk operasi yang lambat, tidak perlu real-time response, atau tidak boleh memblokir user experience.

// Sync: pengirim menunggu
result = paymentService.charge(user, amount)   // block sampai selesai
sendEmail(user, "Payment success")

// Async: pengirim tidak menunggu
paymentQueue.publish({ user, amount })         // langsung return
// worker di background yang process dan kirim email
Analogi: sync kayak order makanan dan nunggu di kasir sampai makanan jadi. Async kayak ambil nomor antrian, duduk, dan dipanggil pas makanan siap.

H.2 Message Queues

Message queue adalah sistem yang menyimpan pesan sementara antara producer dan consumer. Ia memungkinkan dua komponen berkomunikasi tanpa harus online bersamaan dan membuat sistem lebih toleran terhadap lonjakan traffic.

// Alur message queue
[Producer / API]
      |
      | publish("order.created", { orderId: 123 })
      v
[Message Queue (RabbitMQ / Kafka / SQS)]
      |
      | subscribe("order.created")
      v
[Consumer / Worker]
  - kirim email konfirmasi
  - kurangi stok
  - update laporan penjualan
  • RabbitMQ: cocok untuk task queue dan routing yang fleksibel.
  • Kafka: cocok untuk event streaming dengan volume tinggi dan replay.
  • AWS SQS: managed queue, cocok di ekosistem AWS.
Analogi: message queue itu kayak kotak surat — pengirim bisa titip pesan kapan pun, penerima ambil dan baca saat siap.

H.3 Background Jobs

Background job adalah pekerjaan yang dieksekusi di luar siklus request-response utama — biasanya dipicu oleh schedule atau event. Ini penting untuk pekerjaan berat yang tidak perlu hasilnya segera.

// Contoh background job
[Scheduler / Cron]
  - "setiap pukul 00:00": hitung laporan harian
  - "setiap 5 menit": cek order yang pending > 30 menit

[Event-triggered Job]
  - setelah user register: kirim email verifikasi
  - setelah pembayaran berhasil: generate invoice PDF
  - setelah upload: resize dan compress gambar
Analogi: background job kayak petugas kebersihan yang kerja malam — tidak mengganggu aktivitas siang, tapi penting agar sistem tetap rapi.

BAB I. Reliability

I.1 Fault Tolerance

Fault tolerance adalah kemampuan sistem untuk tetap berfungsi meski salah satu komponennya gagal. Sistem yang fault tolerant dirancang dengan asumsi bahwa kegagalan pasti terjadi — bukan jika, tapi kapan.

  • Redundancy: duplikasi komponen kritis (database replica, multi-region).
  • Graceful degradation: fitur non-kritis dimatikan saat beban tinggi.
  • Bulkhead pattern: isolasi kegagalan agar tidak menyebar ke seluruh sistem.
Analogi: pesawat punya dua mesin bukan karena satu tidak cukup, tapi karena kalau satu mati, pesawat tetap bisa mendarat dengan selamat.

I.2 Failover

Failover adalah proses otomatis memindahkan traffic ke sistem cadangan saat sistem utama gagal. Ini memastikan availability tinggi tanpa intervensi manual.

// Database failover
Primary DB  ──replication──►  Replica DB
     |                              |
     | (primary down)               |
     └── health check detects ──────┘
         → traffic dipindahkan ke replica
         → replica jadi primary baru
         → alert dikirim ke on-call engineer

// Zero-downtime target
RTO (Recovery Time Objective)  : berapa lama sistem boleh down
RPO (Recovery Point Objective)  : berapa banyak data boleh hilang
Analogi: failover kayak genset otomatis — begitu listrik PLN mati, genset langsung nyala tanpa harus pencet tombol manual.

I.3 Retry and Circuit Breaker

Retry mengulangi request yang gagal dengan harapan masalahnya sementara. Circuit breaker mencegah sistem terus memanggil service yang sedang bermasalah — ini melindungi dari cascade failure.

// Retry dengan exponential backoff
attempt 1: wait 1s  → gagal
attempt 2: wait 2s  → gagal
attempt 3: wait 4s  → berhasil
(max 3 attempts, jitter ditambah agar tidak semua client retry bersamaan)

// Circuit Breaker — 3 state
CLOSED   : normal, request diteruskan
OPEN     : service gagal melebihi threshold → langsung return error
HALF-OPEN: test dengan sebagian request → kalau sukses, kembali CLOSED

// Threshold contoh
failure rate > 50% dalam 10 detik → circuit OPEN
setelah 30 detik → coba HALF-OPEN
Analogi: circuit breaker kayak sekring listrik di rumah — kalau arusnya berlebihan dan bisa merusak, sekring putus dulu sebelum ada komponen yang terbakar.

BAB J. Performance

J.1 Latency and Throughput

Latency adalah waktu yang dibutuhkan untuk menyelesaikan satu request. Throughput adalah berapa banyak request yang bisa diproses per satuan waktu. Keduanya saling mempengaruhi dan perlu dioptimasi bersama.

// Latency breakdown — contoh request ke API
DNS lookup         :  5ms
TCP handshake      :  10ms
TLS handshake      :  20ms
Server processing  :  80ms
  - auth check     :  5ms
  - DB query       :  60ms
  - JSON serialize :  15ms
Network transfer   :  15ms
──────────────────────────
Total              :  130ms

// Throughput
1 server: 500 RPS (Request Per Second)
10 server horizontal: ~5000 RPS
Analogi: latency kayak berapa lama satu paket sampai ke tujuan. Throughput kayak berapa paket yang bisa dikirim dalam satu jam.

J.2 Query Optimization

Query yang lambat adalah salah satu penyebab paling umum dari performa sistem yang buruk. Optimasi dimulai dari query plan, lalu lanjut ke index, caching, dan akhirnya schema redesign.

-- Cek query plan
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 123 AND status = 'pending';

-- Masalah umum: full table scan
Seq Scan on orders (cost=0.00..4500.00 rows=3 width=100)
  Filter: (user_id = 123)

-- Setelah tambah index
Index Scan using idx_orders_user_id on orders
  (cost=0.42..8.46 rows=3 width=100)

-- N+1 query problem — hindari ini
// Buruk: 1 query untuk daftar user + N query untuk tiap order
users = db.query("SELECT * FROM users")
for user in users:
  orders = db.query("SELECT * FROM orders WHERE user_id = " + user.id)

-- Solusi: gunakan JOIN
SELECT u.*, o.*
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
Analogi: query tanpa index kayak cari nama di buku telepon tanpa urutan alfabet — harus baca satu per satu dari awal.

BAB K. Security

K.1 Authentication and Authorization

Authentication memverifikasi siapa kamu. Authorization memverifikasi apa yang boleh kamu lakukan. Keduanya berbeda dan harus diimplementasi secara terpisah dan berlapis.

// Authentication flow (JWT)
POST /auth/login
  body: { email, password }
  → verify credentials
  → generate JWT: { userId, role, exp }
  → return access_token + refresh_token

// Authorization check
GET /admin/reports
  Authorization: Bearer <token>
  → decode token: { userId: 42, role: "admin" }
  → check: role == "admin" ? proceed : 403 Forbidden

// Role-Based Access Control (RBAC)
admin   → read, write, delete
editor  → read, write
viewer  → read only
Analogi: authentication kayak scan KTP di pintu masuk — membuktikan identitas. Authorization kayak level akses badge karyawan — KTP sama tapi tidak semua bisa masuk ruang server.

K.2 Data Protection

Data protection mencakup enkripsi data saat transit (in-transit) dan saat disimpan (at-rest), serta penanganan data sensitif seperti password, nomor kartu, dan data pribadi.

// Enkripsi in-transit
HTTPS / TLS 1.3 untuk semua komunikasi
HSTS header: Strict-Transport-Security: max-age=31536000

// Password hashing — JANGAN simpan plain text
// Gunakan bcrypt / Argon2
hash = bcrypt.hash(password, saltRounds=12)
isValid = bcrypt.verify(inputPassword, storedHash)

// Sensitive data — jangan log atau expose
// Buruk:
log.info("User login", { email, password })
// Baik:
log.info("User login", { userId, email })

// Data masking di response
{
  "cardNumber": "****-****-****-4242",
  "cvv": "***"
}
Analogi: data protection kayak brankas berlapis — kunci berbeda untuk masuk gedung, masuk ruangan, dan buka brankas. Kalau satu kunci bocor, tidak semua terkena.

BAB L. Monitoring

L.1 Logging

Log adalah catatan kejadian yang terjadi di sistem. Log yang baik mencakup timestamp, severity level, context (request ID, user ID), dan pesan yang cukup informatif untuk debug tanpa perlu membuka kode.

// Level log
DEBUG   : detail teknis untuk development
INFO    : event normal — user login, order created
WARN    : sesuatu tidak ideal tapi sistem masih jalan
ERROR   : sesuatu gagal dan perlu diinvestigasi
FATAL   : sistem tidak bisa lanjut, perlu restart

// Structured log (JSON) — lebih mudah di-query
{
  "timestamp": "2026-03-17T10:30:00Z",
  "level": "ERROR",
  "requestId": "abc-123",
  "userId": 42,
  "message": "Payment failed",
  "error": "Gateway timeout after 5000ms",
  "service": "payment-service"
}
Analogi: log itu kayak buku harian kapal — tiap kejadian dicatat agar kalau ada masalah, bisa ditelusuri balik kapan dan di mana mulai ada yang aneh.

L.2 Metrics and Alerts

Metrics adalah angka yang mencerminkan kondisi sistem dari waktu ke waktu. Alert memberi tahu tim ketika angka tersebut melebihi batas yang dianggap normal, sebelum user merasakan dampaknya.

// Kategori metrik penting (RED Method)
Rate        : berapa request per detik masuk
Errors      : berapa persen request yang gagal
Duration    : berapa lama rata-rata request diproses

// Contoh alert rules
ALERT HighErrorRate
  IF error_rate > 5% for 5 minutes
  NOTIFY on-call via PagerDuty

ALERT SlowAPI
  IF p99_latency > 2000ms for 3 minutes
  NOTIFY team-backend via Slack

// Tools populer
Prometheus + Grafana  : metrics & dashboard
Datadog               : all-in-one observability
AWS CloudWatch        : monitoring di ekosistem AWS
Analogi: metrics dan alert itu kayak dashboard dan lampu indikator di kokpit pesawat — pilot tidak cek semua instrumen satu per satu, tapi kalau ada yang merah, langsung tindakan.

BAB M. Deployment

M.1 CI/CD Basics

CI (Continuous Integration) adalah praktik mengintegrasikan kode secara sering dan menjalankan test otomatis setiap ada perubahan. CD (Continuous Delivery / Deployment) adalah kelanjutannya — perubahan yang lolos test bisa dikirim ke production secara otomatis atau semi-otomatis.

// Pipeline CI/CD umum
[Push code ke Git]
      |
      v
[CI: Build + Test]
  - unit test
  - integration test
  - code quality scan
  - security scan
      |
      v (kalau semua pass)
[CD: Deploy ke Staging]
      |
      v (approval manual atau otomatis)
[Deploy ke Production]
      |
      v
[Smoke test di production]
[Monitor error rate & latency]
Analogi: CI/CD kayak jalur produksi pabrik — setiap produk dicek kualitasnya sebelum dikirim ke toko, bukan dicek setelah keluhan pelanggan datang.

M.2 Containers and Cloud

Container (Docker) mengemas aplikasi beserta semua dependency-nya agar bisa dijalankan secara konsisten di environment mana pun. Kubernetes mengatur container dalam skala besar. Cloud providers seperti AWS, GCP, dan Azure menyediakan infrastruktur yang bisa di-scale sesuai kebutuhan.

// Dockerfile sederhana
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY ./publish .
EXPOSE 8080
ENTRYPOINT ["dotnet", "MyApp.dll"]

// Kubernetes Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-api
  template:
    spec:
      containers:
        - name: my-api
          image: myregistry/my-api:v1.2.0
          ports:
            - containerPort: 8080
Analogi: container itu kayak kontainer kapal cargo — isi apa pun di dalamnya, ukuran dan cara angkutnya sama. Kubernetes kayak pelabuhan yang atur berapa kontainer jalan, mana yang perlu ganti, dan mana yang crash.

BAB N. Trade-offs

N.1 CAP Theorem

CAP Theorem bilang bahwa sistem terdistribusi hanya bisa menjamin dua dari tiga properti secara bersamaan: Consistency, Availability, dan Partition Tolerance. Di dunia nyata, network partition tidak bisa dihindari — jadi pilihan sesungguhnya adalah antara CP dan AP.

CAP Triangle

        Consistency
            /\
           /  \
          /    \
         /  CP  \
        /--------\
       /    CA    \
      /------AP----\
Availability  Partition Tolerance

CP (Consistency + Partition Tolerance):
  - Data selalu konsisten, tapi bisa return error saat network split
  - Contoh: HBase, Zookeeper, etcd

AP (Availability + Partition Tolerance):
  - Selalu bisa diakses, tapi data bisa sementara tidak konsisten
  - Contoh: Cassandra, DynamoDB, CouchDB

CA (Consistency + Availability):
  - Hanya bisa dijamin kalau tidak ada network partition
  - Praktis: sistem single node atau LAN yang terpercaya
Analogi: kayak tiga teman yang harus selalu kompak, tapi kalau HP-nya mati satu, dua yang lain harus pilih: tunggu dia konfirmasi dulu (CP) atau lanjut tanpa dia (AP).

N.2 Consistency vs Availability

Dalam sistem terdistribusi, strong consistency memastikan semua node melihat data yang sama pada saat yang sama, tetapi ini membutuhkan koordinasi yang menambah latency. Eventual consistency memungkinkan data berbeda sementara waktu antar node, tetapi sistem tetap tersedia dan responsif.

// Strong Consistency
User A update stok: 100 → 99
Semua server harus setuju dulu sebelum return sukses
Latency: lebih tinggi, tapi data selalu akurat

// Eventual Consistency
User A update stok: 100 → 99
Return sukses segera, replikasi ke server lain dalam beberapa detik
Latency: rendah, tapi user B mungkin lihat stok = 100 sesaat

// Contoh use case
Strong  : saldo rekening bank, stok obat, tiket pesawat
Eventual: like count, view count, rekomendasi feed
Analogi: strong consistency kayak meeting seluruh tim sebelum keputusan. Eventual consistency kayak update pengumuman di papan tiap divisi — semua akhirnya tahu, tapi tidak persis bersamaan.

N.3 Cost vs Performance

Performa lebih tinggi hampir selalu membutuhkan biaya lebih besar — lebih banyak server, storage yang lebih cepat, CDN premium, atau cache layer tambahan. Keputusan desain harus mempertimbangkan apakah peningkatan performa sepadan dengan biaya tambahannya.

// Contoh trade-off
Skenario: query DB lambat (500ms)

Opsi A: Tambah index
  + Gratis, query → 20ms
  - Tulis jadi sedikit lebih lambat

Opsi B: Tambah Redis cache
  + Query → 1ms
  - Biaya Redis instance (~$50/bln), perlu handle cache invalidation

Opsi C: Upgrade ke DB instance lebih besar
  + Query → 50ms tanpa ubah kode
  - Biaya naik 3x (~$300/bln lebih mahal)

// Decision framework
Tanya dulu: "Seberapa parah ini untuk user?"
Kalau user tidak terasa bedanya → pilih opsi paling murah
Analogi: beli mobil Ferrari buat komisaris perusahaan mungkin worth it. Buat kurir yang butuh mobilnya kuat dan mudah servis, Toyota Kijang jauh lebih masuk akal.

BAB O. Real Examples

O.1 URL Shortener

URL shortener adalah sistem yang mengubah URL panjang menjadi URL pendek dan unik, lalu melakukan redirect ketika URL pendek diakses. Ini contoh bagus untuk belajar hashing, database design, dan caching.

// Functional requirements
- User input URL panjang → dapat URL pendek (e.g. short.ly/abc123)
- User akses URL pendek → redirect ke URL asli
- Optional: analytics (jumlah klik, lokasi)

// High-level design
POST /shorten
  body: { longUrl: "https://..." }
  → generate shortCode (hash atau counter)
  → simpan: { shortCode, longUrl, createdAt } di DB
  → return: { shortUrl: "short.ly/" + shortCode }

GET /{shortCode}
  → lookup shortCode di cache Redis
  → kalau miss, cari di DB
  → return HTTP 301 Redirect ke longUrl

// Estimasi
100 juta URL tersimpan
Read/Write ratio = 100:1
Storage: 100M x ~500 bytes = ~50 GB
Analogi: kayak tukang buat kartu nama singkat untuk alamat website yang panjang — yang penting orang bisa sampai ke tujuan.

O.2 Chat System

Sistem chat real-time membutuhkan koneksi persisten antara client dan server, penyimpanan pesan yang efisien, dan pengiriman notifikasi yang cepat. Ini contoh bagus untuk belajar WebSocket, message queue, dan fan-out pattern.

// Components
[Client A] ─── WebSocket ──► [Chat Server]
[Client B] ─── WebSocket ──► [Chat Server]

// Flow: A kirim pesan ke B
1. A kirim message via WebSocket ke Chat Server
2. Server simpan pesan ke DB (message store)
3. Server publish event ke Message Queue
4. Worker baca event, cek apakah B online
5a. B online  → push via WebSocket
5b. B offline → simpan notifikasi, push saat online

// Penyimpanan pesan
- Time-series atau key-value store (Cassandra, HBase)
- Partition by conversation_id
- Index by timestamp untuk pagination pesan lama
Analogi: chat system kayak sistem walkie-talkie dua arah — kalau penerima sedang off, pesan dititipkan dulu ke towernya dan dikirim begitu nyala.

O.3 E-commerce System

Sistem e-commerce adalah contoh paling lengkap karena mencakup hampir semua komponen: auth, product catalog, cart, order, payment, dan notifikasi. Ini juga menuntut konsistensi data yang tinggi — terutama di bagian stok dan pembayaran.

// Core services (microservices)
User Service      → register, login, profile
Product Service   → catalog, search, detail
Cart Service      → add/remove item (Redis per user)
Order Service     → checkout, order history
Payment Service   → charge, refund, status
Inventory Service → stok management, reservation
Notification Service → email, push, SMS

// Critical flow: checkout
1. Cart service validasi item masih tersedia
2. Inventory service reserve stok (soft lock)
3. Order service buat order dengan status "pending"
4. Payment service proses pembayaran
5a. Sukses → order status "confirmed", stok dikurangi
5b. Gagal  → order dibatalkan, stok dibebaskan

// Teknologi pendukung
Elasticsearch  : product search
Redis          : cart, session, rate limiting
Kafka          : event bus antar service
PostgreSQL     : orders, users, produk
S3             : product images
Analogi: e-commerce system itu kayak mal besar — ada banyak toko terpisah (service), tapi ada sistem kasir terpusat, keamanan, dan pusat informasi yang bikin semuanya jalan mulus sebagai satu pengalaman belanja.

BAB P. Summary

P.1 Key Takeaways

Kalau ditarik benang merahnya, system design itu soal milih kompromi paling masuk akal berdasarkan kebutuhan bisnis dan kondisi teknis. Gak ada desain yang 100% sempurna buat semua situasi. Yang penting adalah lo ngerti kenapa sebuah keputusan diambil, apa dampaknya, dan bagaimana mitigasi risikonya.

  • Mulai dari requirement yang jelas sebelum lompat ke teknologi.
  • Bedakan kebutuhan fungsional dan non-fungsional sejak awal.
  • Rancang high-level architecture sebelum mikirin detail implementasi.
  • Gunakan cache, queue, dan load balancing saat scale mulai naik.
  • Jaga reliability dengan retry, failover, dan circuit breaker.
  • Monitor sistem secara aktif pakai log, metrics, dan alert.
  • Pahami trade-off CAP, consistency, availability, cost, dan performance.
// Mindset cepat pas design interview
1. Clarify requirements
2. Estimate scale
3. Sketch high-level architecture
4. Dive into bottlenecks
5. Discuss trade-offs and failure scenarios
6. Close with monitoring and future improvements
Analogi: system design itu kayak nyusun strategi buka usaha. Lo harus paham target pasar, modal, operasional, dan risiko. Bukan cuma mikirin nama brand atau dekor toko.
