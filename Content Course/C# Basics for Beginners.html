CHAPTER A. Introduction to CSharp

A.1 What is C# and Its History

C# (pronounced C-Sharp) is a modern programming language from Microsoft that runs on the .NET platform. It was born around 1999-2000 to compete with Java, designed to be modern, type-safe, and perfect for the Windows ecosystem.

The name "C#" comes from musical notation, meaning a half-step higher than C. Symbolically, it's one level above C and C++. It's powerful for building business apps, Web APIs, desktop apps, and even games via Unity.

// The very first C# code (version 1.0, 2002) looked like this:
using System;

class HelloWorld
{
    static void Main()
    {
        Console.WriteLine("Hello, World!");
    }
}
Analogy: C# is like a new shoe brand that emerged to fill a gap—older players were too complex (C++) and others were too locked into one ecosystem (Java). C# hit the sweet spot.

A.2 The Creator: Anders Hejlsberg

The main architect of C# is Anders Hejlsberg, a brilliant programmer from Denmark who joined Microsoft in 1996. Before this, he created Turbo Pascal and was the lead designer for Delphi.

He designed C# based on his vast experience to solve programmer frustrations, like manual memory management and vendor lock-in.

  • Automatic Garbage Collector: You don't have to manually manage memory.
  • Type-safe: Data type errors are caught during compilation, not runtime.
  • Multi-paradigm: Supports OOP, functional, and low-level coding if needed.
Analogy: Anders is like a master chef who has cooked in every type of kitchen. He knows exactly why certain recipes fail, so his new recipe avoids all those past mistakes.

A.3 Evolution of C#

C# evolves rapidly. It's not a stagnant language—every 1-2 years, a new version drops, adding features to make code shorter, safer, and more expressive.

  • C# 3.0 (2007): Introduced LINQ, lambda expressions, and var keywords.
  • C# 5.0 (2012): Brought async/await, revolutionizing asynchronous programming.
  • C# 9.0+ (2020s): Brought records, top-level statements, minimal APIs, and global usings.
// Modern C# (version 10+) feels much more concise
// Top-level statement — no need for class Program + Main anymore
using System;

Console.WriteLine("Modern C#, no boilerplate!");
Analogy: C# is like a smartphone that gets an OS update every year. The core remains the same, but it gets smoother and packed with more features. Older knowledge stays relevant while you enjoy the new perks.

A.4 Ecosystem and Usage

.NET used to be tied exclusively to Windows, but now it's fully cross-platform. You can develop and deploy on Windows, Linux, and macOS without issues.

  • .NET SDK: Used for building and running applications.
  • NuGet: The package manager for dependencies.
  • ASP.NET Core: The high-performance framework for web backend and Web API development.
// Example of a minimal API endpoint today
app.MapGet("/ping", () => "pong");
Analogy: It used to be a local shop open in just one city. Now it's a massive franchise with branches everywhere, all sharing the same high standards and fast service.

A.5 Installing Tools

Minimal setup so you can start coding right away:

  1. Install the latest stable .NET SDK.
  2. Pick your IDE: Visual Studio (heavy duty) or VS Code (lightweight and fast).
  3. Test it out by creating and running a new console project.
dotnet new console -n HelloCSharp
cd HelloCSharp
dotnet run
Analogy: Before you start cooking, make sure the stove, gas, and all your kitchen tools are actually set up and ready to go.

CHAPTER B. Basic Syntax and Structure

B.1 Structure of a CSharp Program

Modern C# programs can be super short (using top-level statements), but the core concept remains the same: there's a namespace, a class, and an entry point.

Top-level statements are a game changer. You can write your logic directly in the file, and the compiler handles the background structure. It makes the code cleaner and the learning curve smoother.

  • Using statements: Import required tools, like System.
  • Top-level statements: Run directly at the entry point without extra wrappers.
  • Implicit class: The compiler automatically builds a background class container for you.
  • Global using: (C# 10+) Imports a namespace across your entire project.
using System;
Console.WriteLine("Program started");
Console.WriteLine("Modern C# with no boilerplate!");

// Behind the scenes, the compiler generates:
// [CompilerGeneratedClass].Main(args)
Analogy: A minimalist house still needs a foundation, roof, and doors. Modern C# automatically builds the foundation for you, so you can focus on the interior (your code logic).

B.2 Main Method and Program Entry

In the classic version, a console app starts from the Main() method. It's the very first part the runtime executes.

In production apps, the Main method is crucial for catching errors and loading configurations (like passing connection strings or modes via arguments).

  • Static: Means the method can be called directly without creating an object first.
  • Void: Used if returning nothing. Use int for an exit code (0=success).
  • string[] args: Catches command line arguments as an array of strings.
  • Async Main: (C# 7.1+) Allows you to use async/await for asynchronous setups.
class Program
{
    // Classic Main method - the traditional entry point
    static void Main(string[] args)
    {
        Console.WriteLine("Entered Main");
        
        // Handling command line parameters
        if (args.Length > 0)
        {
            Console.WriteLine($"Arguments: {string.Join(", ", args)}");
        }
    }
    
    // Async Main (C# 7.1+)
    static async Task Main(string[] args)
    {
        Console.WriteLine("Async main started");
        await Task.Delay(1000); // Simulating an async operation
        Console.WriteLine("Async main completed");
    }
}
Analogy: The Main method is your front gate. All program traffic must enter through here first.

B.3 Namespaces and Comments

Namespaces group your code so class names don't crash into each other. Comments provide vital context without stating the obvious.

  • Namespace: Groups related classes together to prevent naming conflicts.
  • Single-line comment: Uses // for short notes.
  • Multi-line comment: Uses /* */ for blocks of text or disabling code.
namespace LearnBasics;

// This is a single-line comment
/*
   This is a multi-line comment
*/
Analogy: A namespace is like a zip code. People can have the exact same name, but different zip codes keep their addresses completely clear.

B.4 Basic Input and Output

Console I/O is fundamental for testing simple logic, debugging, or interacting with a user.

  • Console.Write(): Writes text without moving to a new line.
  • Console.WriteLine(): Writes text and moves to the next line.
  • Console.ReadLine(): Captures the user's input until they press Enter.
  • String interpolation: Uses $"" to easily inject variables into text.
  • Null-coalescing: Uses ?? to provide a default fallback value if something is null.
Console.Write("Your name: ");
string name = Console.ReadLine() ?? "Anon";
Console.WriteLine($"Hello, {name}");

// Handling numerical input safely
Console.Write("Enter age: ");
string ageInput = Console.ReadLine();
if (int.TryParse(ageInput, out int age))
{
    Console.WriteLine($"Next year you'll be {age + 1}");
}
else
{
    Console.WriteLine("Invalid age format!");
}
Analogy: It's like talking to a cashier. The cashier asks a question (Console.Write), you respond (Console.ReadLine), the cashier processes it, and gives you a receipt (Console.WriteLine).

CHAPTER C. Variables and Basic Logic

C.1 Variables and Data Types

Variables are where you store data. In C#, there are two main types that store data differently:

  • Value type: Data is stored directly (e.g., int, double, bool, char).
  • Reference type: Stores only the "address" to the data location (e.g., string, object, class).
int age = 24;
double height = 172.5;
bool active = true;
char grade = 'A';
string name = "Raka"; // This is a reference type
Analogy: A value type is like physical cash in your wallet. A reference type is like holding a locker key; the actual item is inside the locker (memory).

C.2 Type Casting and var Keyword

Casting is used when you need to change data types. Sometimes it's safe and automatic, sometimes you have to force it.

  • Implicit: Safe and automatic. No data is lost (e.g., int to double).
  • Explicit: Risk of data loss, so you must explicitly declare it using parentheses.
  • var: A shortcut for writing data types; the compiler automatically guesses it from the value.
int number = 10;
double implicitCast = number; // Safe

double price = 99.9;
int explicitCast = (int)price; // Truncated to 99

var city = "Jakarta"; // Automatically inferred as string
Analogy: Moving boxes from a small pickup truck to a massive semi-truck (implicit) will definitely fit. But moving from a semi-truck back to a pickup (explicit) means you have to "drop" some boxes to make them fit.

C.3 Operators

Operators are symbols for calculations, comparisons, or injecting logic (and/or).

int a = 8;
int b = 3;

int total = a + b; // Basic math
bool isGreater = a > b; // Comparison (true/false)
bool valid = (a > 0) && (b > 0); // Logical AND
a++; // Increment (add 1)
Analogy: Operators are the tools in your workshop. You have wrenches for assembling (arithmetic) and tape measures for checking (comparison).

C.4 Control Flow

This is the essence of your program, making decisions based on runtime conditions.

int score = 78;

if (score >= 85)
{
    Console.WriteLine("Got an A!");
}
else if (score >= 70)
{
    Console.WriteLine("Got a solid B");
}
else
{
    Console.WriteLine("Got to retake this class");
}
Analogy: Just like a parking attendant directing cars: "If the front slot is empty, park there (if), otherwise check the basement (else if), and if both are full, say parking is closed (else)."

C.5 Looping

Looping is how you avoid manually repeating the exact same tasks and tiring yourself out.

// for: you know how many times to repeat
for (int i = 1; i <= 3; i++)
{
    Console.WriteLine($"Repeating number {i}");
}

// while: repeats as long as the condition is true
int n = 0;
while (n < 2)
{
    Console.WriteLine("Still spinning...");
    n++;
}
Analogy: Like a barista making 20 cups of black coffee. Same machine, same beans, just loop the process 20 times until done.

CHAPTER D. Methods and Collections

D.1 Declaring Methods and Return Types

A method is a specific block of code you can call over and over from anywhere. It makes your codebase modular.

  • Use void if the method just runs an action without returning anything back.
  • Use a data type (int, string, bool) if the method needs to return a result when it's done.
static void Greet(string name)
{
    Console.WriteLine($"Hello, {name}!");
}

static int Add(int x, int y)
{
    return x + y; // Returns the calculated result
}
Analogy: A method is like Standard Operating Procedure (SOP). You write down the guide to mop the floor once. Later, you just say "Run the mopping SOP" without teaching it from scratch.

D.2 Overloading and Optional Parameters

You want your functions to be flexible. Sometimes they need fewer parameters, sometimes you want extra settings.

  • Overloading: Methods can share the same name as long as their parameters (amount or type) are different.
  • Optional parameter: You can assign a default value to a parameter, so it isn't mandatory when calling the method.
// Overloading (same name, different params)
static int Calculate(int a, int b) => a + b;
static int Calculate(int a, int b, int c) => a + b + c;

// Optional parameter (repeat defaults to 1)
static void PrintText(string text, int repeat = 1)
{
    for (int i = 0; i < repeat; i++)
        Console.WriteLine(text);
}

// Can be called with 1 argument, or 2
PrintText("Hello"); 
PrintText("Hello again", repeat: 3);
Analogy: Ordering a "Chicken Meal". The name is the same (overloading), but you can order 1 piece or 2. Hot sauce is an optional parameter—it's off by default, but you can request it.

D.3 Arrays and Multidimensional Arrays

Use arrays when dealing with fixed-size data containers (they can't be resized easily on the fly).

// 1D Array
int[] numbers = { 10, 20, 30 };

// 2D Array (Multidimensional) like a coordinate grid
int[,] matrix =
{
    { 1, 2 }, // Row 0
    { 3, 4 }  // Row 1
};

Console.WriteLine(matrix[1, 0]); // Calls row index-1, col index-0 (result: 3)
Analogy: An array is like a wooden shoe rack that's nailed together. It has exactly 5 slots; it won't stretch to fit 6 pairs. Multidimensional arrays are like gym lockers (rows and columns).

D.4 List and Dictionary

These are modern ways to store data in C#, much more flexible than classic arrays.

  • List<T>: Like an elastic array. You can `.Add()` or `.Remove()` items anytime.
  • Dictionary<TKey, TValue>: Stores key-value pairs. Looking up data using a key is incredibly fast!
// Using List: Flexible size
var nameList = new List { "Alice", "Bob" };
nameList.Add("Charlie"); // Super easy to add items

// Using Dictionary: Key and Value (keys must be unique)
var examScores = new Dictionary
{
    ["Alice"] = 90,
    ["Bob"] = 85
};

// Easy lookup
Console.WriteLine(examScores["Alice"]); // Output: 90
Analogy: A list is an endless queue of people that grows and shrinks over time. A dictionary is a college attendance book; the linkage is guaranteed (Student ID as the key -> Person's Name as the value).

BAB E. OOP and Design Foundation

E.1 Classes, Objects, Properties, Fields, Constructors

This is the foundation of Object-Oriented Programming (OOP). Understanding this well is the key to writing structured and manageable code.

  • Class: Think of this as a blueprint or recipe. A class defines the structure (what data it has) and behavior (what it can do) of something. It doesn't do anything by itself, it's just a definition.
  • Object: This is the real thing created from the blueprint (class). If `Student` is a class, then `studentBudi` and `studentAni` are two different objects, each with their own data (ID, Name).
  • Field: These are variables inside a class that store internal data. The practical rule: fields are always made `private` to protect the data. Example: `_id`.
  • Property: This is the "gate" to access data from outside the class. Properties give you control. You can add validation when data is changed (setter) or format data when it's read (getter). Auto-properties (`public string Name { get; set; }`) are shortcuts when there's no extra logic.
  • Constructor: This is a special method called when a new object is created (`new Student(...)`). Its job is to initialize the object's starting state, making sure the object is born valid.
public class Student
{
    // Field (private, for internal state)
    private readonly string _id;
    private string _name;

    // Property (public, for controlled access)
    public string Name
    {
        get { return _name.ToUpper(); } // Logic when read: always uppercase
        set
        {
            if (!string.IsNullOrWhiteSpace(value)) // Logic when set: cannot be empty
            {
                _name = value;
            }
        }
    }
    
    // Auto-property (shortcut for simple field + property)
    public string Major { get; set; }

    // Constructor
    public Student(string id, string name, string major)
    {
        // Ensuring the object is built in a valid state
        _id = id;
        _name = name;
        Major = major;
        Console.WriteLine($"Student object {_name} with ID {_id} created.");
    }
}

// Creating an object (instantiation)
var student1 = new Student("24001", "budi", "Informatics");
Console.WriteLine(student1.Name); // Output: BUDI
student1.Name = "  "; // Won't change the name because it's invalid
Console.WriteLine(student1.Name); // Output: BUDI
Analogy: A class is a cake recipe. Fields are the raw ingredients you keep in the kitchen (private). Properties are how you serve the cake to guests (public get) and the rules when guests want to add toppings (public set). The constructor is the first process of mixing the dough according to the recipe. The object is the finished cake, ready to eat.

E.2 Encapsulation

Encapsulation is the principle of wrapping data (fields) and the methods that operate on that data into a single unit (class), then hiding unnecessary implementation details from the outside world. The goal is to protect data integrity and simplify class usage.

You don't give direct access to internal data. Instead, you provide public methods or properties as the only official way to interact with the object. This lets you add validation, logging, or other logic without breaking code that uses the class.

public class Wallet
{
    // Internal state, hidden from the outside
    private decimal _balance;

    public Wallet(decimal initialBalance)
    {
        _balance = initialBalance > 0 ? initialBalance : 0;
    }

    // Official way to up balance (with validation)
    public void TopUp(decimal amount)
    {
        if (amount <= 0)
        {
            Console.WriteLine("Top up must be greater than zero.");
            return;
        }
        _balance += amount;
        Console.WriteLine($"Top up {amount:C} successful.");
    }

    // Official way to withdraw balance (with validation)
    public bool Withdraw(decimal amount)
    {
        if (amount <= 0)
        {
            Console.WriteLine("Invalid withdrawal amount.");
            return false;
        }
        if (amount > _balance)
        {
            Console.WriteLine("Insufficient balance.");
            return false;
        }
        _balance -= amount;
        Console.WriteLine($"Withdrawal {amount:C} successful.");
        return true;
    }

    // Official way to view balance (read-only)
    public decimal GetBalance()
    {
        return _balance;
    }
}

// Usage:
var myWallet = new Wallet(50000);
// myWallet._balance = 1000000; // Error! Cannot be accessed directly.
myWallet.TopUp(20000);
myWallet.Withdraw(100000); // Fails, insufficient balance
Console.WriteLine($"Remaining balance: {myWallet.GetBalance():C}");
Analogy: Encapsulation is like an ATM machine. You can't just open the money vault inside (private field). You have to use the provided interface: insert your card, enter your PIN, choose a menu (public methods). The ATM has internal rules (validation) to make sure you can't withdraw more than your balance.

E.3 Inheritance, Polymorphism, Abstraction

These are the three magic pillars of OOP that work together to create flexible and reusable code.

Inheritance: The mechanism where a class (child/derived class) can inherit properties and methods from another class (parent/base class). This creates an "is-a" relationship. Example: `Cat` is an `Animal`. The goal is reusability, so you don't have to rewrite the same code in many classes.

Abstraction: The process of hiding complex implementation details and only showing essential functionality. In C#, this is done through `abstract class` and `interface`. The goal is to simplify. Users of the class don't need to know "how" it works, just "what" it can do.

Polymorphism: The ability of an object to take many forms. In practice, this means you can refer to an object of a child class using a variable of the base class or interface. This lets you write more generic and flexible code that can work with various types of objects as long as they follow the same "contract".

// Abstraction: Defining the "Animal" concept without details
public abstract class Animal
{
    public string Name { get; set; }
    
    // Abstract method, must be implemented by its children
    public abstract void MakeSound();

    // Regular method, can be inherited directly
    public void Sleep()
    {
        Console.WriteLine($"{Name} is sleeping...");
    }
}

// Inheritance: Cat and Dog are "Animals"
public class Cat : Animal
{
    public override void MakeSound() // Providing specific implementation
    {
        Console.WriteLine("Meow!");
    }
}

public class Dog : Animal
{
    public override void MakeSound() // Providing specific implementation
    {
        Console.WriteLine("Woof woof!");
    }
}

// Polymorphism in action
var animalList = new List
{
    new Cat { Name = "Tom" },
    new Dog { Name = "Spike" }
};

// You don't care if it's a Cat or a Dog. You just know they are "Animals"
// and all Animals can "MakeSound".
foreach (var animal in animalList)
{
    Console.Write($"{animal.Name} says: ");
    animal.MakeSound(); // Will call the appropriate method (Meow/Woof woof)
}
Analogy: Imagine `Vehicle` (abstract class). All vehicles have a `Move()` method (abstraction). `Car` and `Motorcycle` are `Vehicle` (inheritance). On the road (List<Vehicle>), we see them all as "vehicles", but each moves in its own way (polymorphism).

E.4 Interfaces and Abstract Classes

Both are used for abstraction, but have different purposes and use cases. Choosing the right one is very important in software design.

Interface:

  • Defines a contract. It only contains definitions for methods, properties, events, or indexers, with no implementation at all.
  • A class can implement many interfaces. This solves the "multiple inheritance" problem in C#.
  • Creates a "can-do" relationship. Example: `Plane` and `Bird` can both implement `ICanFly`.
  • Very ideal for Dependency Injection and decoupling.

Abstract Class:

  • Defines a basic framework for a closely related group of classes.
  • Can contain methods with implementation (real code) and also `abstract` methods that must be overridden.
  • A class can only inherit from one abstract class.
  • Creates an "is-a" relationship. Example: `Cat` and `Dog` are `Animal`.

When to Use Which?

  • Use an abstract class if you have a group of classes that are very similar and want to share the same implementation code.
  • Use an interface if you want to define a capability that can be owned by completely unrelated classes.
  • Good design principle: "Program to an interface, not an implementation." This means your code should depend on abstractions (interface/abstract class), not concrete classes. This makes the system more flexible and easier to maintain.
// Interface defines a "capability"
public interface IPayment
{
    bool Pay(decimal amount);
    string CheckStatus();
}

// Unrelated classes can share the same capability
public class CreditCard : IPayment
{
    public bool Pay(decimal amount) { /* payment gateway logic */ return true; }
    public string CheckStatus() { return "CreditCard: Paid"; }
}

public class EWallet : IPayment
{
    public bool Pay(decimal amount) { /* deduct e-wallet balance logic */ return true; }
    public string CheckStatus() { return "E-Wallet: Paid"; }
}

// Our code depends on the abstraction (IPayment), not the specifics
public class Cashier
{
    public void ProcessPayment(IPayment method, decimal total)
    {
        Console.WriteLine("Processing payment...");
        if (method.Pay(total))
        {
            Console.WriteLine(method.CheckStatus());
        }
    }
}

// Much more flexible
var cashier = new Cashier();
cashier.ProcessPayment(new CreditCard(), 50000);
cashier.ProcessPayment(new EWallet(), 25000);
Analogy: An interface is like a USB port. It defines the connection standard, but doesn't care what device is plugged in (mouse, flash drive, fan). An abstract class is like a car chassis. All sedans from one brand might share the same chassis, but the body and engine can be different.

CHAPTER F. Error Handling and Data Processing

F.1 Exception Handling (try-catch-finally)

Exception handling is the mechanism for handling errors that occur while the program is running (runtime errors). Without this, the app will crash immediately when it hits an unexpected problem, like dividing by zero or a missing file.

  • `try`: This block contains code that might cause an error.
  • `catch`: This block runs if an exception happens in the `try` block. You can catch specific exception types for different handling.
  • `finally`: This block always runs, whether there's an error or not. It's the ideal place to clean up resources, like closing database or file connections.

Best Practice: Catch the most specific exception first. Don't get used to catching `Exception` (the most general type) unless you really want to handle all errors the same way—which usually isn't a good idea.

try
{
    Console.WriteLine("Trying to read a file...");
    string content = File.ReadAllText("non-existent-file.txt");
    Console.WriteLine(content);
}
catch (FileNotFoundException ex)
{
    // Specific handling if the file is not found
    Console.WriteLine($"Specific error: File not found. Details: {ex.Message}");
}
catch (Exception ex)
{
    // General handling for any other unexpected errors
    Console.WriteLine($"A general error occurred: {ex.Message}");
}
finally
{
    // This block always runs, perfect for cleanup
    Console.WriteLine("Process finished, finally block executed.");
}
Analogy: `try` is like walking on a slippery floor. `catch` is your readiness to keep your balance if you slip (catching `FileNotFoundException`). `finally` is what you always do afterward, whether you made it safely or not, like cleaning your shoes.

F.2 Custom Exception

Making your own exception type (custom exception) is super useful for signaling errors that are specific to your app's business domain. This lets higher-level code (the caller) give a more relevant response than just showing a generic error message.

Custom exceptions are usually made by inheriting from the `Exception` class.

// 1. Create your own custom exception class
public class InsufficientBalanceException : Exception
{
    public decimal CurrentBalance { get; }
    public decimal WithdrawalAmount { get; }

    public InsufficientBalanceException(string message, decimal balance, decimal withdrawal) : base(message)
    {
        CurrentBalance = balance;
        WithdrawalAmount = withdrawal;
    }
}

// 2. Use it in business logic
public class BankAccount
{
    private decimal _balance = 1000;

    public void Withdraw(decimal amount)
    {
        if (amount > _balance)
        {
            // Throw the specific exception with rich context
            throw new InsufficientBalanceException(
                "Withdrawal failed due to insufficient balance.", 
                _balance, 
                amount);
        }
        _balance -= amount;
        Console.WriteLine($"Successfully withdrew {amount}. Remaining balance: {_balance}");
    }
}

// 3. Catch it at the UI or service level
var account = new BankAccount();
try
{
    account.Withdraw(1500);
}
catch (InsufficientBalanceException ex)
{
    Console.WriteLine($"[!] Transaction Failed: {ex.Message}");
    Console.WriteLine($"    Your Balance: {ex.CurrentBalance}, You tried to withdraw: {ex.WithdrawalAmount}");
}
Analogy: Instead of a doctor just saying "You're sick" (generic error message), a custom exception is like the doctor giving a specific diagnosis: "You have the flu, here's the prescription." The message is clear and the action can be more precise.

F.3 File I/O (Input/Output)

Interacting with files is a common need. C# provides several ways, from simple to more controlled.

Simple Way (`File` class): Great for quick operations on small files. The `File` class automatically manages opening and closing files.

// Write all text to a file (will overwrite if exists)
File.WriteAllText("notes.txt", "This is the first line.\nThis is the second line.");

// Read all text from a file
string fileContent = File.ReadAllText("notes.txt");
Console.WriteLine(fileContent);

// Append text to an existing file
File.AppendAllText("notes.txt", "\nThis is an added line.");

Controlled Way (`StreamWriter`/`StreamReader`): Good for big files or when you need more control, like writing line by line. It's important to make sure the stream is closed properly using a `using` block.

// Write to a file line by line
using (StreamWriter writer = new StreamWriter("log.txt"))
{
    writer.WriteLine("Log started...");
    writer.WriteLine($"Timestamp: {DateTime.Now}");
    // 'using' will automatically call writer.Close() at the end of the block
}

// Read from a file line by line
using (StreamReader reader = new StreamReader("log.txt"))
{
    string line;
    while ((line = reader.ReadLine()) != null)
    {
        Console.WriteLine($"-- {line}");
    }
}
Analogy: The `File` class is like buying instant food—fast and practical. `StreamWriter` is like cooking from scratch, it takes more process but you have full control over every step and ingredient.

F.4 LINQ (Language-Integrated Query)

LINQ is a revolutionary feature in C# that lets you query various data sources (in-memory collections, databases, XML) with a uniform and expressive syntax, similar to SQL.

With LINQ, code for filtering, sorting, and transforming data becomes much shorter and easier to read than manual loops.

Common Examples of Using LINQ:

public record Product(string Name, string Category, int Price);

var productList = new List
{
    new Product("Gaming Laptop", "Electronics", 20000),
    new Product("Plain T-Shirt", "Fashion", 150),
    new Product("Wireless Mouse", "Electronics", 300),
    new Product("Jeans", "Fashion", 400),
    new Product("24-inch Monitor", "Electronics", 2500)
};

// 1. Filtering (Where): Get all electronic products
var electronics = productList.Where(p => p.Category == "Electronics");

// 2. Sorting (OrderBy): Sort products from most expensive
var expensiveProducts = productList.OrderByDescending(p => p.Price);

// 3. Projection (Select): Just grab the names of the products
var productNames = productList.Select(p => p.Name);

// 4. Chaining: Combining multiple operations
// Get names of electronics under 1000, sorted by name
var queryResult = productList
                    .Where(p => p.Category == "Electronics" && p.Price < 1000)
                    .OrderBy(p => p.Name)
                    .Select(p => p.Name);

// Execute query and display results
foreach (var name in queryResult)
{
    Console.WriteLine(name); // Output: Wireless Mouse
}
Analogy: Imagine you have a huge stack of documents. Without LINQ, you have to check each one manually (for loop). With LINQ, you just tell your assistant: "Please find all documents about 'project X' (Where), sort them by newest (OrderBy), and just give me the titles (Select)." Way more efficient.

CHAPTER G. Async and Project Structure

G.1 Async and Await Basics

Asynchronous programming is one of the most important concepts in modern C#, especially for responsive apps like UI and web servers. The goal is to run time-consuming operations (I/O-bound) without blocking the main thread.

The Problem: Synchronous (Blocking) Operations

Without async, when your app calls a slow process (like downloading a file or querying a database), the main thread stops and waits until it's done. In desktop apps, this makes the UI "Not Responding". On web servers, it means the server can't handle other requests, drastically lowering throughput.

The Solution: `async` and `await`

  • `async`: This keyword is added to a method signature to indicate the method can run `await` operations inside. Async methods usually return `Task` or `Task`.
  • `await`: This keyword "pauses" method execution until the awaited `Task` is done. While waiting, the main thread is returned to the pool and is free to do other work. When the `Task` finishes, the method continues from where it left off.
// Method that simulates a slow I/O process (e.g. calling an API)
static async Task FetchDataFromApiAsync()
{
    Console.WriteLine("Starting to fetch data from API...");
    // await 'frees' the thread for 2 seconds, the app doesn't freeze
    await Task.Delay(2000); 
    Console.WriteLine("Data successfully fetched from API.");
    return "This is the downloaded data";
}

// Usage in Main method (or another async method)
Console.WriteLine("Before calling the async method.");
// Call and wait for the result without blocking
string result = await FetchDataFromApiAsync(); 
Console.WriteLine("After calling the async method.");
Console.WriteLine($"Result received: {result}");

// Output:
// Before calling the async method.
// Starting to fetch data from API...
// (app waits for 2 seconds here, but it doesn't freeze)
// Data successfully fetched from API.
// After calling the async method.
// Result received: This is the downloaded data

Important: `async` and `await` are best suited for I/O-bound operations (network request, file access, database query). For CPU-bound operations (heavy mathematical calculations), use `Task.Run` to offload it to a separate thread.

Analogy: You order coffee at a cafe (`await GetDataAsync`). The barista starts making your coffee (I/O process). Instead of standing at the counter doing nothing until it's ready (blocking), you can sit and play on your phone (thread is free). When your name is called because the coffee is ready (task done), you go back to the counter to pick it up (execution continues).

G.2 Understanding Project Files and Build Run Process

The `.csproj` (C# Project) file is the heart of every .NET project. It's an XML file that tells the `dotnet` compiler everything about your project: how to compile, what dependencies are needed, and what output to produce.

General Structure of a `.csproj` File:

<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <!-- Output type: Exe (application), Library (DLL) -->
    <OutputType>Exe</OutputType>
    
    <!-- Target Framework: .NET version used -->
    <TargetFramework>net8.0</TargetFramework>
    
    <!-- Enables modern C# features like implicit usings -->
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

  <ItemGroup>
    <!-- List of dependencies from NuGet -->
    <PackageReference Include="Newtonsoft.Json" Version="13.0.3" />
  </ItemGroup>

</Project>

Build and Run Flow:

  1. `dotnet restore`: This command reads the `` section in the `.csproj` file and downloads all needed libraries from NuGet to the local cache.
  2. `dotnet build`: This command compiles all `.cs` files in your project into Intermediate Language (IL) files and saves them as `.dll` or `.exe` in the `bin/Debug/` folder.
  3. `dotnet run`: This command is a shortcut that automatically runs `build` (if there are changes) and then executes the output.
Analogy: The `.csproj` file is like a recipe and shopping list for a chef. `PropertyGroup` is the cooking instructions (bake at 180°C, dish type: cake). `ItemGroup` is the list of ingredients to buy (flour, eggs, JSON library). `dotnet restore` is shopping for ingredients, `dotnet build` is cooking, and `dotnet run` is serving and eating the cake.

H.1 ASP.NET Core Basics

Analogy: If learning C# is like learning cooking techniques (chopping, sautéing, roasting), then learning ASP.NET Core is like opening a restaurant. You start using all those techniques in an organized system (a commercial kitchen) to serve many customers (HTTP requests).

H.2 Web API Fundamentals

A Web API is a digital "waiter" that lets different applications (e.g. a React frontend, a mobile app, or another server) talk to each other through the standard HTTP protocol. This communication happens through "endpoints".

Common HTTP Verbs (Methods):

  • `GET`: To request/get data. (Example: `/products` to get all products).
  • `POST`: To send/create new data. (Example: `/products` with new product data in the request body).
  • `PUT`: To completely update existing data.
  • `DELETE`: To delete data.

In modern ASP.NET Core (Minimal APIs), creating an endpoint is incredibly easy:

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// GET Endpoint: Fetch all products
app.MapGet("/products", () => 
{
    return new[] { "Keyboard", "Mouse", "Monitor" };
});

// GET by ID Endpoint: Fetch one product by ID
app.MapGet("/products/{id}", (int id) => 
{
    // Logic to find a product by id...
    return Results.Ok($"Fetching product with ID: {id}");
});

// POST Endpoint: Create a new product
app.MapPost("/products", (Product newProduct) => 
{
    // Logic to save the new product to the database...
    return Results.Created($"/products/{newProduct.Id}", newProduct);
});

app.Run();

public record Product(int Id, string Name);
Analogy: A Web API is like the menu and ordering system in a restaurant. `GET /menu` is you asking for the menu. `POST /orders` is you handing a new order to the kitchen. The waiter (HTTP) and the standard language (JSON) ensure the customer and kitchen can communicate without misunderstandings.

H.3 MVC Pattern High Level

MVC (Model-View-Controller) is an architectural design pattern that separates an application into three main components. Its goal is "Separation of Concerns", which makes the code much more organized, easier to test, and easier to maintain.

  • Model: Represents the data and business logic of the app. It's the "brain". The Model knows nothing about the UI or how data is displayed. It's only responsible for valid data and rules. (Example: `Product`, `User` classes, and validation methods).
  • View: Responsible for displaying data to the user. It's the "face" of the app (UI). In a Web API, the "View" is often just data formatted as JSON, not an HTML page.
  • Controller: Acts as the middleman between the Model and the View. It receives input from the user (HTTP requests), processes it by calling the Model, and then picks the right View to return the result.

The simple workflow: Request enters the Controller -> Controller calls the Model to fetch or manipulate data -> Model returns data back to the Controller -> Controller passes that data to the View (JSON) to send back as the response.

Analogy: Imagine the process in a restaurant. The customer (User) gives an order to the Waiter (Controller). The Waiter writes it down and gives it to the Chef (Model). The Chef cooks according to the order (business logic). Once cooked, the Chef gives the food to the Waiter. The Waiter then serves it on a nice plate (View) to the customer's table. Every part has its own specific job.

H.4 Learning Path After Fundamentals

Mastering C# fundamentals is just the first step. To become a competent backend developer, you still have a long journey ahead. Here is a realistic and structured roadmap:

  1. Build a Full CRUD Project: Build a simple Web API from start to finish that can Create, Read, Update, and Delete (CRUD) a single entity (e.g. Products or Notes). Deploy this project to a platform like Azure or another hosting service. This is the ultimate test of understanding.
  2. Learn Entity Framework Core: This is the standard Object-Relational Mapper (ORM) in .NET. EF Core lets you interact with databases using C# objects instead of writing raw SQL. Learn about `DbContext`, migrations, and LINQ to Entities queries.
  3. Implement Authentication & Authorization: Secure your API. Learn how JSON Web Tokens (JWT) work for stateless authentication. Implement a role-based (Admin, User) and policy-based authorization system.
  4. Understand Testing and Logging: Code without tests is fragile. Learn how to write Unit Tests to test business logic in Models, and Integration Tests to test your API endpoints as a whole. Implement structured logging (e.g., using Serilog) to monitor your app in production.
  5. Level Up Architecture: Once comfortable with MVC, start learning cleaner architectural patterns like Clean Architecture or Onion Architecture. This completely distances your business logic from the framework, making the app highly maintainable and testable.
Analogy: Learning fundamentals is like getting your driver's license. You know how to drive. The next step is driving in the inner city (CRUD project), learning navigation (EF Core), understanding traffic signs (Auth), learning car maintenance (Testing/Logging), and finally planning cross-country road trips (Clean Architecture).
