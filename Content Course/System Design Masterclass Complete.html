System Design Masterclass Complete

A complete guide to mastering System Design from scratch to interview-ready at Big Tech, explained in a casual but "heavy-duty" tone.

Quick Map Section:

BAB A. System Design Fundamentals

A.1 Introduction to System Design

System Design is the process of designing system architecture so it can handle millions of users without "exploding." It's not about writing code; it's about choosing the right "Lego bricks" like databases, servers, load balancers, and caches to work together seamlessly.

Analogy: System Design is like planning a city map. You have to think about where to put highways (load balancers), where to place supermarkets (databases), and how the logistics flow so everything doesn't jam up during rush hour (traffic spike).
  • Focus on high-level architecture (HLD).
  • Learn to choose trade-offs: there is no perfect design, only the right design for the requirement.
  • Crucial for senior engineer interviews and designing robust real-world applications.

A.2 System Design Template

To avoid getting overwhelmed, you need a framework. Don't start drawing diagrams right away. Start with the basics so your design doesn't have holes.

Analogy: Like ordering food at a stall. You need to know how hungry you are (requirements), what your budget is (capacity), what you want to eat (API design), and only then does the vendor sort your plate (HLD).
  1. Functional Requirements: What are the main features? (e.g., can upload photos).
  2. Non-functional Requirements: How fast? How secure? High availability? (Latency, Availability, Consistency).
  3. Capacity Estimation: Back-of-the-envelope calculations for traffic and storage.
  4. API Design: Define the communication endpoints between components.
  5. High-level Design: Draw the big boxes and data flows.
  6. Deep Dive: Dissect the most crucial parts (e.g., how to keep the feed fast).

BAB B. Core Concepts (Building Blocks)

B.1 Requirements & Estimation

Before jumping straight into coding, you must know how big your system will be. Without calculations, you're just guessing.

Distinguish between features (Functional) and qualities (Non-functional). Calculate "Daily Active Users" (DAU), throughput (RPS), and future storage needs.

Analogy: Like planning a wedding. You need to know who's invited (Functional), the food must be delicious and warm (Non-functional), and how many portions to prepare (Estimation).
// Example Rough Capacity Estimation
DAU: 10 Million Users
Write per day (1 post/user): 10 Million
Storage per post: 500 KB
Total Storage/Day: 10M * 500KB = 5 TB
Total Storage/Year: 5TB * 365 = ~1.8 PB

B.2 API Design

APIs are how your components talk. REST or gRPC? What matters is consistency, readability, and security. Don't let your endpoints get messy like a tangled cable box.

Analogy: An API is like a menu at a cafe. If you want to order coffee, there's a name, price, and options (cold/hot). The server (waiter) understands the request because the menu is clear.
  • REST API: Uses HTTP methods like GET, POST, PUT, DELETE.
  • Idempotency: Ensure that if a user double-taps "Pay" due to lag, they aren't charged twice.
  • Query Params: Use for filtering and pagination to keep responses lightweight.

B.3 High-Level Design (HLD)

In this stage, you draw the "Big Picture." Don't worry about specific libraries yet. Think about how the Client connects to the Load Balancer, then the API Gateway, Service, and finally the Database.

Analogy: HLD is like an MRT map. You only see which station connects where. You don't care about the color of the train's bolts; you care that the path is clear.
[Client] -> [Load Balancer] -> [API Gateway] -> [Microservices] -> [Database/Cache]

B.4 Data & Storage

Choose the right "warehouse" for your data. Not everything belongs in SQL. Sometimes NoSQL makes more sense for unconventional data or extreme performance needs.

Analogy: SQL is like a tidy bookshelf (categorized); NoSQL is like a laundry basket (input anything fast, retrieval is flexible).
  • SQL (PostgreSQL, MySQL): Best for strong relations and secure transactions (e.g., bank balance).
  • NoSQL (MongoDB, Cassandra): Best for massive scale and flexible schemas.

BAB C. Real World System Design Cases

C.1 Social Media System (Instagram / Newsfeed)

The biggest challenge in building a feed is the "Fan-out." If a popular user posts, how does it instantly reach millions of follower screens?

Analogy: A feed is like a morning newspaper. There's a printing team (Service), delivery agents (Message Queue), and the porch where it's dropped (Cache/Feed Store).
  • Feed Generation: Can be "Pull" (on-request) or "Push" (pre-computed).
  • Media Storage: Use S3 for files and a CDN for global speed.
  • Optimization: Use Redis for the most recent feeds to avoid heavy database queries.

C.2 Video Streaming System (YouTube / Netflix)

Video is extremely heavy. You must slice it into small chunks with multiple resolutions so it streams smoothly from old phones up to 4K TVs.

Analogy: Like selling meatballs. If the meatball is the size of a basketball (raw video), people can't eat it. It must be sliced (transcoding/chunking) for easy consumption (streaming).
  • Video Pipeline: Upload -> Transcoding -> Storage -> CDN.
  • Adaptive Bitrate Streaming: Automatically downsizes quality if the user's signal is weak.

C.3 URL Shortener (TinyURL)

This is a mapping problem. How do you convert a 1000-character URL into just 6-8 unique characters?

Analogy: Like a queue number at a clinic. The short number "A-01" represents your long full name and medical data.
  • Encoding: Use Base62 (0-9, a-z, A-Z) to get massive combinations with few characters.
  • Hashing vs Counter: Use MD5 prefixes or an encoded auto-increment DB ID.

C.4 Messaging System (WhatsApp)

Chat revolves around real-time speed and correct ordering. Messages must arrive fast and in the exact sequence they were sent.

Analogy: Like talkies. You stay tuned to the frequency (WebSocket) to hear the other person the moment they speak.
  • WebSocket: Allows the server to "whisper" messages to the user's phone instantly (Push).
  • Online/Offline Status: Measured using a system "heartbeat."
  • Consistency: Use sequence IDs to keep messages orderly.

C.5 Search System

Finding data across millions of rows is impossible with `LIKE %keyword%`. You need intelligent indexing.

Analogy: Like looking for a word in a textbook. Don't read from page one; check the "Index" at the back.
  • Inverted Index: Maps words to the documents that contain them.
  • Ranking: Decides what is most relevant to show at the top.

C.6 Booking System (Airbnb)

The nightmare scenario: "Double Booking." Two people paying for the same room at the exact same second.

Analogy: Like the last seat on a bus. Without a strict conductor (locking mechanism), two people would try to sit in one chair.
  • Concurrency Control: Use Pessimistic or Optimistic Locking.
  • Search: Use Geospatial Indexing for location-based radius searches.

C.7 Notification System

Must deliver millions of notifications via push, email, and SMS without slowing down the main server.

Analogy: Like mall announcements. The PA system (Notification Service) is separate from the cashier (Checkout) so the cashier can focus on the money.
  • Decoupling: Use RabbitMQ/Kafka so sending notifs doesn't block main processes.
  • Rate Limiting: Prevents "attacking" users with 100 notifications a minute.

C.8 Logging System (Distributed Logging)

If your app is on 100 servers, you can't check them one-by-one for errors. Collect them centrally.

Analogy: Like complex security cams. Records are gathered in one guard room (Centralized Storage) so if a thief (bug) shows up, the guard can see all angles at once.
  • Log Aggregator: Asynchronously collects logs from various servers.
  • Searchable: Use the ELK Stack (Elasticsearch, Logstash, Kibana) for easy debugging.

BAB D. System Design Patterns & Concepts

D.1 Rate Limiting

Don't let bots or trolls bomb your server with junk requests. Set a cap!

Analogy: Like a "buy 1 get 1" promo limited to 2 packs per person. Ensures everyone gets some and stock isn't wiped out by scalpers.
  • Token Bucket: User gets "coins" to access. If coins run out, they wait for refilling.
  • Sliding Window: Checks the request count in the last X seconds with high precision.

D.2 Real-Time Communication

Today, a 5-second wait feels like 5 years. Your system must deliver data instantly.

Analogy: Chatting must be like a phone call (WebSocket), not like sending mail (HTTP Polling) where you keep asking "Any mail for me?".

D.3 Scalability Concepts

Horizontal Scaling is key. Instead of one expensive supercomputer, use 10 cheap ones working together.

Analogy: Hauling 100 tons of rice. Instead of one giant truck (Vertical), use 10 smaller ones (Horizontal). If one breaks, 90 tons keep moving.

BAB E. Deep Dive Engineering Decisions

E.1 Database & Data Modeling

Choose the right schema for retrieval. Clean normalization or fast denormalization?

Analogy: Organizing a closet. Normalization involves separating everything: shirts in one drawer, pants in another. Tidy, but putting on an outfit requires opening many drawers (Join). Denormalization is hanging "complete sets" together. Messy if you only want a shirt, but fast when you want a whole outfit.

E.2 Media Handling & Processing

Massive video upload? Don't use the app server directly. Use a Pre-signed URL so the client uploads directly to cloud storage.

Analogy: Like dropping a package at a courier booth (Pre-signed URL) instead of traveling to the central mailing hub. They handle the rest (Pipeline).

E.3 Concurrency & Consistency

How do you keep data accurate when thousands update the same item? This is the "final boss" of system design.

Analogy: Fighting for the TV remote. There must be rules on who holds it (Locking), otherwise the channels change so fast the TV hangs.

BAB F. System Design Interview Preparation

F.1 System Design Thinking Framework

Design interviews aren't about right/wrong answers; they're about how you think. Follow the sequence.

Analogy: Like being asked "How do you organize a massive concert?". You don't discuss stage lights first; you ask about attendance, location, and permits before speakers.

F.2 Common Interview Problems

Spot the patterns. Almost every problem is a variation of Messaging, Feed, Search, or Booking. Master a pattern, and you can answer many questions.

Analogy: Like cooking techniques. If you know how to sauté (Pattern), you can cook greens, veggies, or meat using that same technique.

F.3 Communication & Explanation

Explain your trade-offs. Don't just say "use Kafka because it's cool." Say "use Kafka for high throughput and data persistence."

Analogy: Explaining choices to a partner. Don't just say "I bought this phone because it's good." Say "I bought this for the battery life and the camera quality for our calls." Clearer, right?

BAB G. Engineering Mindset

G.1 Trade-offs in System Design

There's no "best system." There's only "choosing your poison." Fast but delayed data (AP), or accurate data with potential errors (CP).

Analogy: Choosing a home. Near the office but loud/expensive, or far away but quiet/cheap? You can't have it all at once.

G.2 Continuous Learning

Tech changes constantly. System Design is a muscle. Train it daily by reading tech blogs from massive companies or tinkering with architectures.

Analogy: Learning System Design is like learning guitar. You can't just read about it; you have to play (practice) so your fingers get used to the notes.

🚀 Tips

  • Don’t trust your first thought
  • Always ask: “Is this true?”
  • Slow down your thinking when making decisions
  • Focus on clarity, not being right
