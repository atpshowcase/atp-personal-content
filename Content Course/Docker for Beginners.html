BAB A. Introduction to Docker

A.1 What is Docker

Docker itu platform buat bungkus aplikasi beserta dependency-nya ke dalam container, jadi jalan di laptop lo, server orang lain, atau cloud tetap konsisten.

docker --version

docker info
Analogi: Docker itu kayak bungkus makanan lengkap. Bukan cuma makanannya, tapi saus, sendok, dan kemasannya ikut kebawa.

A.2 Why Docker is Used in Modern Development

Docker dipakai karena tim dev sekarang butuh environment yang cepat, konsisten, dan gampang dipindahin. "Works on my machine" jadi jauh lebih jarang kejadian.

docker run hello-world
Analogi: semua orang di tim pakai cetakan yang sama, jadi hasil kue gak beda-beda tiap dapur.

A.3 Problems Docker Solves

Docker nyelesain masalah versi dependency bentrok, setup environment lama, onboarding ribet, dan deployment yang suka beda antara local dan server.

# cek container yang jalan
Docker ps
Analogi: daripada tiap orang ngerakit motor dari nol, Docker kasih motor yang udah siap nyala.

A.4 Virtual Machines vs Containers

VM bawa satu OS penuh, sedangkan container share kernel host. Makanya container lebih ringan dan start-nya jauh lebih cepat.

# container biasanya start dalam hitungan detik
docker run -d nginx
Analogi: VM itu kayak sewa satu rumah penuh, container itu kayak sewa kamar yang tetap punya fasilitas bareng.

A.5 Docker Ecosystem Overview

Ekosistem Docker bukan cuma engine. Ada image, container, volume, network, registry, Compose, sampai orchestration.

docker image ls
docker container ls
docker volume ls
docker network ls
Analogi: Docker ecosystem itu kayak satu kompleks usaha, ada gudang, kendaraan, toko, sama jalur distribusinya.

BAB B. Installing Docker

B.1 Installing Docker on Windows

Di Windows, paling umum pakai Docker Desktop. Biasanya butuh WSL2 atau Hyper-V tergantung setup mesin lo.

# setelah install, cek versi
Docker --version
Analogi: ini kayak pasang mesin utama dulu sebelum buka bengkel.

B.2 Installing Docker on Linux

Di Linux, install bisa lewat package manager distro. Biasanya lebih ringan karena engine jalan native.

sudo apt update
sudo apt install docker.io
sudo systemctl start docker
Analogi: Linux itu kayak motor barebone, pasang part langsung ke rangka utama.

B.3 Installing Docker on macOS

Di macOS, biasanya pakai Docker Desktop juga. Experience-nya cukup mulus buat development harian.

docker --version
docker info
Analogi: tinggal pasang aplikasi utama, lalu cek semua tombol berfungsi.

B.4 Verifying Docker Installation

Setelah install, selalu test pakai image sederhana untuk mastiin engine, CLI, dan akses registry jalan.

docker run hello-world
Analogi: habis servis motor, ya pasti dites jalan dulu di depan bengkel.

B.5 Docker CLI Basics

CLI Docker itu pintu masuk utama. Lo harus nyaman sama command dasar sebelum naik ke Compose atau CI/CD.

docker help
docker run --help
docker ps
Analogi: CLI itu setir dan pedalnya. Kalau ini belum enak, nyetir lebih jauh bakal ribet.

BAB C. Docker Concepts

C.1 What is a Container

Container itu instance hidup dari image yang berjalan sebagai proses terisolasi. Isolasi ini bikin aplikasi lo punya ruang kerja sendiri (filesystem, proses, network) tapi tetap lebih ringan dibanding VM.

Di real project, satu aplikasi sering dipecah jadi beberapa container: misal API, worker, dan database. Masing-masing punya tugas spesifik biar gampang di-scale dan di-debug.

  • Container itu sementara (ephemeral): bisa dibuat, dihentikan, lalu diganti kapan saja.
  • Data penting jangan ditaruh di layer container, simpan di volume.
  • Satu container idealnya jalanin satu proses utama.
docker run -it ubuntu bash
Analogi: image itu cetakan martabak, container itu martabak yang lagi dimasak di kompor.

C.2 What is an Image

Image adalah blueprint read-only yang dipakai buat bikin container. Image disusun dari layer, jadi proses build lebih efisien karena Docker bisa pakai cache layer yang tidak berubah.

Makanya urutan di Dockerfile itu ngaruh banget: kalau lo taruh langkah yang sering berubah di atas, cache gampang invalid dan build jadi lambat.

  • Image tidak "jalan" sendiri, harus dibuat container dulu.
  • Satu image bisa dipakai bikin banyak container.
  • Tag image penting buat kontrol versi yang jelas.
docker pull nginx
docker image inspect nginx
Analogi: image itu resep final yang tinggal dipakai berulang kali.

C.3 Docker Registry

Registry itu tempat simpan dan distribusi image, bisa public atau private. Dari registry inilah hasil build di CI/CD dibagikan ke tim atau server deployment.

Tim biasanya pakai naming convention biar rapih, misal company/service:version. Dengan pola ini, rollback jadi gampang karena versi image jelas.

  • Public registry: cepat buat belajar atau share image umum.
  • Private registry: cocok untuk image internal perusahaan.
  • Jangan lupa autentikasi sebelum push/pull ke registry private.
docker login
docker pull nginx
Analogi: registry itu gudang pusat tempat stok barang disimpen dan diambil.

C.4 Docker Hub

Docker Hub adalah registry paling populer dan jadi pintu masuk banyak image standar. Lo bisa nemu image official dari vendor besar seperti Nginx, Redis, MySQL, sampai Postgres.

Walau praktis, tetap selektif. Pilih image official atau image dengan reputasi baik, lalu pin tag versi biar tidak berubah diam-diam.

  • Cari image dengan label official atau verified publisher.
  • Hindari bergantung penuh ke tag latest.
  • Cek dokumentasi image untuk env var dan default config.
docker search redis
docker pull postgres:16
Analogi: Docker Hub itu marketplace utamanya anak Docker.

C.5 Container Lifecycle

Container punya lifecycle yang jelas: create, start, running, stop, restart, lalu remove. Paham alur ini bikin lo cepat baca kondisi service saat troubleshooting.

Di workflow modern, container sering dianggap disposable. Kalau ada perubahan config besar, biasanya container lama dibuang dan diganti container baru dari image terbaru.

  1. Create: container dibuat dari image tapi belum jalan.
  2. Start: container mulai dieksekusi.
  3. Stop: proses utama berhenti dengan aman.
  4. Remove: container dihapus dari host.
docker create nginx
docker start 
docker stop 
docker rm
Analogi: kayak toko pop-up. Bisa dibuka, ditutup, dibuka lagi, atau dibongkar total.

BAB D. Working with Docker Images

D.1 Searching Images

Sebelum pull image, jangan asal ambil nama pertama yang muncul. Cek dulu reputasi image, dokumentasinya, frekuensi update, dan apakah image itu official atau verified publisher.

Langkah kecil ini penting buat keamanan dan stabilitas aplikasi, karena image yang asal bisa bawa dependency rentan atau konfigurasi yang gak cocok buat production.

  • Prioritaskan image official atau publisher tepercaya.
  • Lihat deskripsi dan cara pakai image sebelum dipakai.
  • Cek tag versi yang tersedia, jangan langsung pakai latest.
docker search node
Analogi: sebelum beli spare part, cek dulu toko resminya atau bukan.

D.2 Pulling Images from Docker Hub

Pull artinya ngambil image dari registry ke mesin lokal. Best practice-nya adalah selalu pakai tag versi spesifik supaya environment lo deterministik dan gampang diulang.

Kalau pakai latest, isi image bisa berubah kapan saja tanpa lo sadar. Ini sering bikin bug yang susah dilacak karena hari ini jalan, besok beda perilaku.

  1. Tentukan versi image yang jelas sesuai kebutuhan app.
  2. Pull image versi itu ke local environment.
  3. Gunakan versi yang sama di CI/CD dan production.
docker pull node:20-alpine
Analogi: daripada bilang "kasih apa aja", mending sebut ukuran dan model yang jelas.

D.3 Listing Local Images

Image lokal perlu dicek rutin supaya lo tau mana yang dipakai aktif, mana yang dangling, dan mana yang cuma bikin disk cepat penuh.

Ini kebiasaan sederhana tapi berdampak besar, terutama kalau lo sering build ulang atau gonta-ganti branch yang menghasilkan banyak image sementara.

  • Pantau ukuran image buat identifikasi yang paling boros storage.
  • Bedakan image aktif vs image sisa eksperimen.
  • Gunakan filter untuk mempermudah audit image tertentu.
docker image ls
docker image ls --filter "reference=node*"
Analogi: cek stok gudang rutin biar gak penuh barang nganggur.

D.4 Removing Images

Image yang gak dipakai sebaiknya dibersihin, apalagi kalau build berkali-kali dan layer numpuk. Cleanup rutin bikin mesin dev lebih lega dan proses kerja tetap gesit.

Hati-hati saat hapus image yang masih dipakai container aktif. Biasakan cek dulu container yang berjalan sebelum pruning besar-besaran.

  • Hapus image spesifik kalau memang tidak dipakai.
  • Pakai prune untuk bersih-bersih image dangling.
  • Jangan cleanup agresif saat environment sedang dipakai tim bersama.
docker rmi node:20-alpine
docker image prune
Analogi: kardus bekas yang udah gak kepakai jangan dibiarkan numpuk di pojok.

D.5 Image Tagging

Tag dipakai buat nandain versi image, dan ini kunci release yang reproducible. Dengan tag yang rapi, lo bisa tahu image mana yang dipakai di dev, staging, dan production.

Strategi tagging yang umum dipakai tim adalah gabungan semantic version, nama branch, atau short commit SHA supaya jejak build kebaca jelas.

  1. Gunakan tag versi manusiawi, contoh v1.0.0.
  2. Tambahkan tag teknis bila perlu, contoh commit hash.
  3. Simpan pola tag yang konsisten lintas pipeline.
docker tag myapp:latest myapp:v1.0.0
docker tag myapp:latest myapp:main-a1b2c3d
Analogi: tag itu kayak label batch produksi, biar gak ketukar barang lama dan baru.

BAB E. Working with Containers

E.1 Running a Container

Command paling dasar ya `docker run`. Dari sini lo bisa atur port, env, volume, network, dan nama container.

docker run -d --name web -p 8080:80 nginx
Analogi: ini kayak buka lapak baru lengkap sama alamat tokonya.

E.2 Listing Running Containers

Lo harus sering cek container yang aktif, terutama pas local dev atau debug service yang banyak.

docker ps
docker ps -a
Analogi: lihat toko mana yang lagi buka dan mana yang lagi tutup.

E.3 Stopping Containers

Kalau service udah gak perlu jalan, stop aja supaya resource mesin gak kebuang.

docker stop web
Analogi: lampu toko mati kalau tokonya udah tutup, biar hemat listrik.

E.4 Removing Containers

Container yang udah selesai dipakai bisa dihapus supaya daftar container tetap bersih.

docker rm web
Analogi: tenda event dibongkar setelah acara selesai.

E.5 Viewing Container Logs

Logs itu sumber informasi utama pas container error, restart terus, atau service gak respon.

docker logs web
docker logs -f web
Analogi: log itu CCTV operasional. Kalau ada masalah, lihat rekamannya dulu.

BAB F. Dockerfile

F.1 What is a Dockerfile

Dockerfile adalah blueprint langkah demi langkah untuk ngebangun image secara konsisten. Sekali resep ini beres, siapa pun di tim bisa build hasil yang sama tanpa nebak-nebak setup.

Nilai utamanya ada di reproducibility. Jadi ketika bug muncul di production, lo bisa lacak image dibangun dari resep yang mana dan memperbaikinya secara terukur.

  • Dockerfile menyimpan standar build satu tim.
  • Perubahan environment bisa ditrack lewat version control.
  • Cocok buat local dev, CI pipeline, sampai deployment.
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["npm", "start"]
Analogi: Dockerfile itu resep masakan yang bisa diulang sama persis kapan pun.

F.2 Basic Dockerfile Structure

Struktur Dockerfile yang sehat biasanya dimulai dari base image, lalu folder kerja, copy file, install dependency, expose port, dan command runtime. Urutan ini bukan formalitas, tapi ngaruh ke cache dan kecepatan build.

Prinsipnya, langkah yang jarang berubah taruh di atas. Langkah yang sering berubah (misal source code) taruh belakangan supaya cache layer lebih kepakai.

  1. FROM pilih pondasi image yang tepat.
  2. WORKDIR tetapkan direktori kerja agar command konsisten.
  3. COPY pindahkan file sesuai kebutuhan build/runtime.
  4. RUN jalankan instalasi dependency atau build app.
  5. CMD/ENTRYPOINT tentukan proses utama saat container jalan.
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY publish .
ENTRYPOINT ["dotnet", "MyApp.dll"]
Analogi: susunan bahan, langkah, dan cara nyaji harus runtut.

F.3 Common Dockerfile Instructions

Instruction yang paling sering dipakai: FROM, WORKDIR, COPY, RUN, ENV, EXPOSE, CMD, ENTRYPOINT. Kalau paham fungsi tiap instruction, lo bakal lebih gampang baca Dockerfile tim lain dan cepat debugging.

  • FROM: base image awal.
  • COPY dan ADD: pindah file ke image (umumnya pakai COPY).
  • RUN: eksekusi command saat build.
  • ENV: set variabel lingkungan default.
  • EXPOSE: dokumentasi port aplikasi di container.
  • CMD: command default yang bisa dioverride.
  • ENTRYPOINT: command utama yang lebih rigid.
ENV ASPNETCORE_URLS=http://+:8080
      EXPOSE 8080
      CMD ["dotnet", "MyApp.dll"]
Analogi: ini toolkit dasar yang harus hafal sebelum improvisasi.

F.4 Building Images with Dockerfile

Setelah Dockerfile siap, build image pakai docker build dan selalu kasih tag yang jelas. Tag ini penting buat release management, testing, dan rollback.

Di project tim, proses ini biasanya dijalankan otomatis di CI. Tapi lokal tetap perlu dipahami supaya lo cepat validasi perubahan sebelum push.

  1. Build image dari folder yang berisi Dockerfile.
  2. Kasih nama dan versi yang mudah dikenali.
  3. Jalankan container untuk smoke test cepat.
docker build -t myapp:v1 .
      docker run --rm -p 8080:8080 myapp:v1
Analogi: resep udah siap, tinggal produksi batch pertama.

F.5 Optimizing Dockerfile

Optimasi Dockerfile fokus ke build cepat, image kecil, dan surface area keamanan yang lebih sempit. Ini berpengaruh langsung ke kecepatan deploy dan biaya storage/bandwidth.

Teknik yang paling sering dipakai adalah multi-stage build, pemilihan base image ringan, dan penggunaan dockerignore agar context build tidak kebanyakan sampah file.

  • Pakai base image minimal sesuai kebutuhan runtime.
  • Manfaatkan layer cache dengan urutan instruction yang benar.
  • Gunakan multi-stage build untuk pisahkan proses build dan runtime.
  • Buang dependency/tool yang tidak dibutuhkan di image final.
# contoh .dockerignore
node_modules
bin
obj
.git
# contoh singkat multi-stage
      FROM node:20 AS build
      WORKDIR /app
      COPY package*.json ./
      RUN npm ci
      COPY . .
      RUN npm run build

      FROM nginx:alpine
      COPY --from=build /app/dist /usr/share/nginx/html
Analogi: bawa barang seperlunya saja, jangan satu gudang ikut diangkut.

BAB G. Managing Data with Docker

G.1 Docker Volumes

Volume dipakai buat nyimpen data di luar lifecycle container, jadi data tetap aman walau container dihapus lalu dibuat ulang. Ini penting banget buat database, file upload, dan cache yang harus bertahan lama.

Karena volume dikelola Docker, dia lebih portable dibanding path random di host. Tim biasanya pilih volume untuk data aplikasi yang sifatnya persisten dan dipakai lintas restart.

  • Data tetap ada walau container diganti versi baru.
  • Lebih aman untuk service stateful seperti MySQL dan Postgres.
  • Bisa dipakai ulang oleh container pengganti dengan cepat.
docker volume create mysql_data
docker run -v mysql_data:/var/lib/mysql mysql:8
Analogi: container itu kiosnya, volume itu gudang permanennya.

G.2 Bind Mounts

Bind mount itu map folder host langsung ke container. Ini paling cocok buat local development karena perubahan file di editor langsung kebaca di container tanpa rebuild image.

Tapi bind mount juga punya risiko, karena container bisa akses file host yang di-mount. Jadi di environment production, bind mount biasanya dibatasi atau dihindari kecuali benar-benar perlu.

  1. Gunakan bind mount untuk coding harian dan hot reload.
  2. Pastikan path host jelas dan tidak terlalu luas.
  3. Pisahkan pola mount dev dan prod agar tidak kebawa salah konfigurasi.
docker run -v ${PWD}:/app node:20-alpine
Analogi: kayak buka jendela langsung ke ruang kerja utama, jadi perubahan kelihatan real-time.

G.3 Persisting Data in Containers

Kalau app lo nyimpen upload, database, atau file hasil proses, jangan simpan di filesystem container doang. File di layer container bisa hilang saat container dihapus atau diganti.

Prinsip dasarnya sederhana: container boleh disposable, data jangan disposable. Makanya untuk semua data penting, selalu tentukan strategi persistence dari awal.

  • Simpan data bisnis ke volume atau storage eksternal.
  • Pastikan path data aplikasi tidak nyangkut di folder sementara.
  • Siapkan backup rutin untuk data kritikal.
docker run -d -v uploads:/app/uploads myapp
Analogi: jangan taruh arsip penting di tenda acara yang besok dibongkar.

G.4 Volume Management

Volume juga harus dikelola dengan disiplin: cek isi dan relasinya, bersihkan orphan volume, dan lindungi volume yang menyimpan data penting. Tanpa manajemen, volume gampang numpuk dan bikin storage host cepat habis.

Di tim production, volume management biasanya masuk checklist operasional bareng monitoring, backup, dan kebijakan retensi data.

  1. Audit volume secara berkala untuk tahu mana yang aktif.
  2. Hapus volume orphan dengan hati-hati setelah verifikasi.
  3. Dokumentasikan volume kritikal agar tidak kehapus saat cleanup.
docker volume ls
docker volume inspect mysql_data
docker volume prune
Analogi: gudang juga butuh audit rutin, bukan cuma diisi terus.

BAB H. Docker Networking

H.1 Container Networking Basics

Container punya network sendiri. Supaya bisa diakses dari host atau container lain, lo harus paham mapping dan network mode.

docker network ls
Analogi: ini soal alamat rumah dan jalur antar gang.

H.2 Port Mapping

Port mapping menghubungkan port host ke port container. Tanpa ini, service dalam container belum tentu bisa diakses dari browser lo.

docker run -p 8080:80 nginx
Analogi: kayak pasang papan alamat toko biar customer tahu masuk lewat pintu mana.

H.3 Bridge Network

Bridge network adalah mode default. Container dalam bridge yang sama bisa saling ngobrol pakai nama container.

docker network create app_net
docker run --network app_net --name api myapi
Analogi: satu kompleks ruko yang semua tenant-nya saling kenal alamat internal.

H.4 Host Network

Host network bikin container share network host langsung. Biasanya dipakai di Linux untuk kasus tertentu, tapi harus hati-hati.

docker run --network host nginx
Analogi: toko lo buka langsung di jalan utama tanpa pagar kompleks.

H.5 Container Communication

Kalau banyak service, pastikan komunikasi antar-container jelas: nama service, port internal, dan network yang sama.

docker exec -it api ping db
Analogi: tim operasional harus punya jalur komunikasi internal yang lancar, bukan teriak-teriak beda gedung.

BAB I. Docker Compose

I.1 What is Docker Compose

Compose dipakai buat jalanin banyak container sekaligus lewat satu file konfigurasi.

docker compose up -d
Analogi: daripada buka toko, gudang, kasir satu-satu, Compose nyalain semua sekaligus.

I.2 Installing Docker Compose

Sekarang Compose umumnya udah jadi bagian dari Docker Desktop atau plugin Docker CLI modern.

docker compose version
Analogi: fitur bonus yang udah nempel di paket utamanya.

I.3 Docker Compose File Structure

File Compose biasanya berisi services, volumes, networks, dan environment variables.

services:
  api:
    build: .
  db:
    image: postgres:16
Analogi: blueprint satu event lengkap, siapa isi booth mana dan pakai fasilitas apa.

I.4 Running Multiple Containers

Ini manfaat utama Compose: app, database, redis, worker bisa jalan bareng dengan sekali command.

docker compose up -d
docker compose ps
Analogi: satu saklar buat nyalain satu studio produksi.

I.5 Managing Multi-Service Applications

Compose bikin setup multi-service lebih rapi, terutama buat local dev dan integration testing.

docker compose down
docker compose logs -f
Analogi: semua tenant dalam satu mal bisa diatur dari panel pusat.

BAB J. Docker for Application Development

J.1 Containerizing Applications

Containerizing artinya bikin aplikasi lo bisa jalan di container dengan dependency yang lengkap dan terkontrol.

docker build -t myapp:dev .
docker run -p 5000:5000 myapp:dev
Analogi: aplikasi lo dibikin jadi paket siap kirim, bukan rakitan lepas.

J.2 Docker for Backend Applications

Backend cocok banget di-Docker-in karena gampang dipair sama database, queue, cache, dan API gateway.

docker run -d --name api -p 8080:8080 myapi
Analogi: backend itu dapur utama, Docker bikin dapurnya gampang dipindah lokasi.

J.3 Docker for Databases

Database di container enak buat local dev dan testing, tapi harus disiplin soal volume dan backup.

docker run -d --name pg -e POSTGRES_PASSWORD=secret -p 5432:5432 postgres:16
Analogi: database itu brankas. Tempatnya bisa portable, tapi isi dan kuncinya harus dijaga.

J.4 Development vs Production Containers

Container development biasanya lebih fleksibel, sedangkan production harus kecil, aman, dan deterministik.

# dev
Docker compose up

# prod
docker run --read-only --restart always myapp:v1
Analogi: baju rumah beda sama baju meeting penting. Fungsinya sama, standarnya beda.

J.5 Environment Variables

Env var dipakai buat konfigurasi runtime seperti connection string, API key, dan mode environment.

docker run -e ASPNETCORE_ENVIRONMENT=Development myapp
Analogi: env var itu setting panel di belakang layar, bukan tulisan yang dipajang di depan toko.

BAB K. Docker with Databases

K.1 Running MySQL in Docker

MySQL sering dipakai untuk local dev, demo, atau integration test. Jangan lupa set password dan volume.

docker run -d --name mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  -p 3306:3306 \
  -v mysql_data:/var/lib/mysql \
  mysql:8
Analogi: mesin kasir bisa dipindah-pindah, tapi data transaksi harus tetap aman.

K.2 Running PostgreSQL in Docker

Postgres populer buat banyak app modern. Setup dasarnya juga sederhana di Docker.

docker run -d --name postgres \
  -e POSTGRES_PASSWORD=secret \
  -p 5432:5432 \
  postgres:16
Analogi: sama kayak buka cabang baru, tinggal sambungin mesin inti dan jalur datanya.

K.3 Running MongoDB in Docker

MongoDB juga gampang dijalankan dalam container, apalagi buat local development.

docker run -d --name mongo -p 27017:27017 mongo:7
Analogi: gudang dokumen fleksibel yang raknya gampang diatur.

K.4 Connecting Applications to Containers

App lo harus tahu hostname, port, user, password, dan network yang benar supaya bisa connect ke database container.

# contoh compose hostname
Server=db;Port=5432;Database=appdb;
Analogi: nomor telepon, alamat, dan kode akses harus bener semua baru nyambung.

K.5 Data Persistence

Tanpa persistence, restart container bisa bikin data lenyap. Ini kesalahan klasik anak baru Docker.

docker volume create pg_data
Analogi: jangan simpan kas warung di meja lipat yang bisa kebalik kapan aja.

BAB L. Docker for .NET Applications

L.1 Containerizing .NET Applications

.NET cocok banget di-container-kan karena build dan runtime bisa dipisah rapi pakai multi-stage build.

docker build -t dotnetapp:v1 .
Analogi: proses produksi dan toko jualannya dipisah, jadi hasil akhir lebih rapi.

L.2 Dockerizing ASP.NET Applications

ASP.NET app biasanya expose port dan jalan lewat kestrel. Jadi perlu base image runtime yang tepat.

EXPOSE 8080
ENTRYPOINT ["dotnet", "MyApi.dll"]
Analogi: lo harus pasang pintu masuk yang jelas buat customer web.

L.3 Multi-Stage Builds

Multi-stage build bikin image akhir lebih kecil karena tool build gak ikut dibawa ke runtime.

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS final
Analogi: peralatan tukang ditinggal di workshop, yang dikirim ke pelanggan cuma produk jadinya.

L.4 Optimizing .NET Docker Images

Optimasi fokus ke publish trimmed kalau cocok, multi-stage, layer caching, dan minimal copy file.

dotnet publish -c Release -o /app/publish
Analogi: kirim paket yang isinya padat dan perlu aja, bukan plus kardus ekstra.

L.5 Running .NET Containers

Setelah build, test jalanin container dan cek endpoint atau health check-nya.

docker run -d -p 8080:8080 dotnetapp:v1
Analogi: habis produksi, barang tetap harus dites sebelum dilempar ke pasar.

BAB M. Docker in CI/CD

M.1 Docker in Continuous Integration

Di CI, Docker dipakai buat bikin build environment konsisten dan gampang direplikasi di pipeline.

docker build -t myapp:${BUILD_ID} .
Analogi: jalur produksi otomatis yang semua step-nya pakai mesin yang sama.

M.2 Building Images in Pipelines

Pipeline biasanya build image lalu kasih tag berdasarkan branch, commit SHA, atau release version.

docker build -t registry/myapp:1.2.0 .
Analogi: setiap batch produksi dikasih nomor seri.

M.3 Pushing Images to Registries

Setelah build lolos, image didorong ke registry supaya siap dipakai deployment.

docker push registry/myapp:1.2.0
Analogi: barang jadi dipindah dari pabrik ke gudang distribusi.

M.4 Deployment with Docker

Deployment bisa sesederhana pull image terbaru lalu restart service, atau masuk ke orchestrator.

docker pull registry/myapp:1.2.0
docker run -d registry/myapp:1.2.0
Analogi: stok baru datang, toko tinggal ganti display barang lama.

M.5 Versioning Docker Images

Versioning penting buat rollback. Jangan terlalu bergantung ke `latest` kalau lo mau deployment yang aman.

myapp:1.2.0
myapp:1.2.1
myapp:main-abc123
Analogi: simpan riwayat versi itu kayak punya arsip nota lama kalau sewaktu-waktu perlu mundur.

BAB N. Docker Security

N.1 Container Security Basics

Container memang ringan, tapi bukan berarti aman otomatis. Security tetap harus dipikirin dari image sampai runtime.

docker inspect myapp
Analogi: pintu rumah ringan bukan berarti gak perlu kunci.

N.2 Image Vulnerabilities

Image bisa punya dependency rentan. Makanya biasakan scan image dan pakai base image tepercaya.

docker scout quickview myapp:latest
Analogi: bahan baku murah belum tentu aman buat dijual ke pelanggan.

N.3 Secure Image Practices

Gunakan image official, pin versi, hapus tool yang gak perlu, dan jalankan app dengan user non-root kalau bisa.

USER appuser
Analogi: kasih akses pegawai secukupnya, jangan semua pegang kunci master.

N.4 Managing Secrets

Jangan hardcode password atau API key di Dockerfile. Pakai env var aman, secret manager, atau platform secret store.

docker run --env-file .env myapp
Analogi: password brankas jangan ditulis di papan pengumuman.

N.5 Best Security Practices

Principle-nya: minim privilege, minimal image, rutin update, scan berkala, dan batasi akses network.

# contoh prinsip
read-only filesystem
non-root user
pinned versions
Analogi: keamanan yang bagus itu kombinasi pagar, kamera, dan kunci, bukan berharap aman doang.

BAB O. Debugging and Troubleshooting

O.1 Inspecting Containers

Inspect dipakai buat lihat detail config container: env, mount, network, state, dan image yang dipakai.

docker inspect web
Analogi: buka data lengkap kendaraan sebelum nebak-nebak kerusakan.

O.2 Viewing Logs

Log biasanya jadi petunjuk tercepat buat cari error startup, port bentrok, atau config salah.

docker logs -f api
Analogi: dengar suara mesin dulu sebelum bongkar total.

O.3 Debugging Container Issues

Kalau container bermasalah, cek image, command, env, volume, network, dan permission satu-satu.

docker exec -it api sh
printenv
Analogi: diagnosa masalah itu step by step, bukan asal ganti semua komponen.

O.4 Performance Monitoring

Monitor CPU, memory, network, dan disk I/O supaya tahu bottleneck container ada di mana.

docker stats
Analogi: lihat konsumsi bensin, suhu mesin, dan performa rem sebelum perjalanan jauh.

O.5 Common Docker Errors

Error umum: port already allocated, permission denied, image not found, connection refused, dan volume/path salah.

Bind for 0.0.0.0:8080 failed: port is already allocated
Analogi: banyak error Docker itu sebenarnya cuma alamat bentrok atau kunci salah.

BAB P. Best Practices

P.1 Writing Efficient Dockerfiles

Tulis Dockerfile yang ringkas, layer-nya masuk akal, dan gampang dibaca tim lain.

# copy file project dulu baru restore dependency
COPY *.csproj .
RUN dotnet restore
Analogi: susun rak gudang biar barang cepat dicari, bukan asal taruh.

P.2 Reducing Image Size

Image kecil bikin pull cepat, deploy cepat, dan attack surface lebih kecil.

FROM node:20-alpine
Analogi: kirim paket kecil lebih murah dan lebih cepat sampai.

P.3 Container Management

Manajemen container itu soal penamaan jelas, log rapi, cleanup rutin, dan pemisahan fungsi.

docker ps -a
docker container prune
Analogi: gudang yang rapih bikin kerja tim lebih cepat.

P.4 Production Best Practices

Untuk production, pastikan restart policy, health check, monitoring, secret handling, dan image versioning jelas.

docker run --restart always myapp:v1
Analogi: toko yang buka setiap hari butuh SOP operasional, bukan cuma semangat awal.

P.5 Docker Project Structure

Project Docker yang bagus biasanya punya Dockerfile jelas, compose terpisah per env, dan file ignore yang rapi.

Dockerfile
docker-compose.yml
.dockerignore
env/
Analogi: lemari arsip yang tertata bikin orang baru cepat paham isi kantor.

BAB S. Glossary

S.1 Container

Container adalah unit aplikasi yang berjalan dari image dan punya runtime sendiri.

Analogi: kios aktif yang sedang buka dan melayani pelanggan.

S.2 Image

Image adalah blueprint atau template untuk membuat container.

Analogi: cetakan atau resep final.

S.3 Dockerfile

Dockerfile adalah resep build image langkah demi langkah.

Analogi: buku resep produksi.

S.4 Volume

Volume adalah storage persisten yang terpisah dari lifecycle container.

Analogi: gudang tetap walau kiosnya pindah.

S.5 Registry

Registry adalah tempat menyimpan dan mendistribusikan image.

Analogi: gudang distribusi pusat.

S.6 Network

Network mengatur bagaimana container saling terhubung atau diakses dari luar.

Analogi: jalan, alamat, dan jalur komunikasi.

S.7 Orchestration

Orchestration adalah cara mengelola banyak container atau service secara terkoordinasi.

Analogi: manajer operasional yang ngatur semua cabang sekaligus.
Kesimpulan: Docker itu bukan sekadar tool buat jalanin container, tapi fondasi penting buat development modern yang konsisten, cepat, dan gampang dipindah dari laptop ke production.
